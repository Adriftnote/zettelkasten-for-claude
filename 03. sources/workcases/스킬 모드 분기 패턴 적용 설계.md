# 스킬 모드 분기 패턴 적용 설계

> SkillNet(REF-090) 저장소 탐색 → PyTorch pr-review 스킬에서 발견한 "모드 분기 + 외부 참조 파일" 패턴을 우리 스킬 시스템에 적용하는 설계.

## 1. 패턴 출처

### PyTorch pr-review 스킬 구조

```
skills/pr-review/
├── SKILL.md              ← 모드 분기 로직 + 공통 원칙 (204줄)
├── review-checklist.md   ← 상세 체크리스트 (외부 참조)
└── bc-guidelines.md      ← 하위 호환성 가이드 (외부 참조)
```

**모드 분기**:
```bash
/pr-review 12345           # PR 번호 모드 → gh CLI로 PR 데이터 fetch
/pr-review branch          # 로컬 브랜치 모드 → git diff main...HEAD
/pr-review 12345 detailed  # 상세 모드 → line-by-line 분석 추가
```

**핵심 원리**:
- SKILL.md는 모드 분기 + 공통 원칙만 담음 (라우터 역할)
- 상세 체크리스트는 별도 파일로 분리 → 필요한 모드에서만 로딩
- Activation 시 컨텍스트 비용 절감 (전체 읽기 → 모드별 선택적 읽기)

### SkillNet에서 확인한 다른 스킬들

| 스킬 | 출처 | 특징 |
|------|------|------|
| pr-review | PyTorch (97K stars) | 모드 분기 + 외부 참조 — **채택** |
| multi-reviewer-patterns | wshobson/agents | Severity 보정 기준표 — 참고만 |
| debugger | awesome-llm-apps | 범용 체크리스트 — `systematic-debugging`이 더 나음 |
| changelog-generator | Composio | 커밋 분류 — 우리에게 불필요 |
| workflow-automation | vibeship | n8n 비교 — 우리 실무 경험이 더 깊음 |

## 3. 설계

### 3.1 rpg — 모드 분기 + 템플릿 분리

**현재 문제**: 442줄 단일 SKILL.md에 4가지 입력 분기 + 3가지 노드 타입(project/module/function) 템플릿이 전부 혼재.

**적용 후 구조**:
```
skills/rpg/
├── SKILL.md                    ← 모드 분기 + 공통 원칙 (~120줄 목표)
└── references/
    ├── project-template.md     ← project 노드 작성 템플릿
    ├── module-template.md      ← module 노드 작성 템플릿
    └── function-template.md    ← function 노드 작성 템플릿
```

**모드 정의**:
```bash
/rpg extract <file|folder>      # rpg-extract CLI → 노트 초안 생성
/rpg update <module-name>       # 기존 RPG 노트 읽기 → 코드 변경사항 반영
/rpg doc                        # 대화 컨텍스트에서 RPG 노드 작성
```

**SKILL.md 구조 (안)**:
```markdown
# RPG (Repository Planning Graph)

## 모드

### /rpg extract <path>
1. `rpg-extract <path> --module-name <name>` 실행
2. 출력에서 노드 타입 판단 (project/module/function)
3. 해당 템플릿 참조: `references/{type}-template.md`
4. basic-memory write_note로 저장

### /rpg update <module-name>
1. basic-memory read_note로 기존 노트 로딩
2. rpg-extract로 현재 코드 상태 추출
3. diff 비교 → 변경된 함수/호출 관계 반영
4. basic-memory edit_note로 업데이트

### /rpg doc
1. 대화 컨텍스트에서 코드 구조 파악
2. 노드 타입 판단 → 해당 템플릿 참조
3. basic-memory write_note로 저장

## 공통 규칙
- project: zettelkasten, folder: 05. code/
- vecsearch 중복 확인 필수
- (기존 공통 규칙 유지)
```

### 3.2 figma-workflow — 모드 분기 + 체크리스트 분리

**현재 문제**: 172줄에 디자인→코드, 토큰 동기화, Variable Manager 조작이 혼재. 토큰만 동기화하려 해도 전체 워크플로우를 읽어야 함.

**적용 후 구조**:
```
skills/figma-workflow/
├── SKILL.md                    ← 모드 분기 + 공통 원칙 (~80줄 목표)
└── references/
    ├── component-workflow.md   ← 디자인→코드 전체 플로우
    ├── token-sync-steps.md     ← 토큰 동기화 절차
    └── variable-manager.md     ← 플러그인 조작 가이드
```

**모드 정의**:
```bash
/figma component <url>          # 디자인→코드 생성 (MCP 전체 플로우)
/figma token-sync               # get_variable_defs → tokens.css 갱신만
/figma variable                 # Variable Manager 플러그인 조작
```

**SKILL.md 구조 (안)**:
```markdown
# Figma Workflow

## 모드

### /figma component <url>
1. URL에서 fileKey, nodeId 추출
2. `references/component-workflow.md` 참조
3. get_design_context → 구현 → get_screenshot 검증

### /figma token-sync
1. `references/token-sync-steps.md` 참조
2. get_variable_defs → tokens.css 갱신
3. 컴포넌트 파일 건드리지 않음 (비파괴 원칙)

### /figma variable
1. `references/variable-manager.md` 참조
2. 플러그인 코드 수정 → Figma 재시작 안내

## 공통 규칙
- Tailwind 사용 금지 → var(--token, fallback)
- 컴포넌트: worker-frontend/figma-components/
- (기존 rules/figma-design-system.md 자동 로딩)
```

## 5. 주의사항

- **기존 동작 깨지면 안 됨** — 인자 없이 `/rpg` 호출하면 현재처럼 대화 기반으로 동작해야 함
- **references/ 파일은 Claude가 모드에 따라 선택적으로 Read** — 전체 로딩 아님
- **frontmatter `argument-hint` 업데이트** — 새 모드 문법 반영
- **commands/ 레거시 충돌 확인** — 동명 commands/ 파일 있으면 삭제 필요

## 6. SkillNet 탐색 위임 패턴

SkillNet 스킬 검색·다운로드·분석은 메인 컨텍스트를 오염시킴 (테이블 출력, 시행착오, SKILL.md 전문). Explore agent에 위임하여 요약만 수신.

**위임 프롬프트 (안)**:
```
SkillNet에서 "{주제}" 관련 스킬을 조사해줘.

1. `skillnet search "{주제}" --mode keyword --limit 5` 실행
2. 평가 Good이 3개+ 인 스킬만 선별
3. `skillnet download <url> --target-dir /tmp/skillnet-samples` 로 다운로드
4. 각 SKILL.md를 읽고 다음을 정리:
   - 스킬 이름, 출처 (프로젝트)
   - 핵심 아이디어 (1-2줄)
   - 우리 시스템에 참고할 만한 패턴이 있는지

우리 환경: Claude Code, .claude/skills/{name}/SKILL.md, 29개 커스텀 스킬.
범용 베스트프랙티스가 아닌 구조적 패턴만 보고해줘.
```

**에이전트**: `Explore` (Read+Bash 가능, 요약 가능) · model: `sonnet` (분석 필요)

**주의**: `skillnet` CLI의 cp949 인코딩 문제 → `PYTHONIOENCODING=utf-8 python -c "subprocess.run(['skillnet', ...])"` 래핑 필요 (Windows)

## 7. Subagent에 SkillNet 검색 위임 패턴

Subagent가 작업 시작 전 SkillNet을 참고 자료로 검색하게 하는 패턴. **항상이 아닌, 오케스트레이터가 판단하여 프롬프트에 포함할 때만.**

### 적용 기준

| 작업 유형 | 검색 지시 | 이유 |
|----------|:---:|------|
| 새 스킬/워크플로우 설계 | ✅ | 구조적 패턴 참고 (모드 분기 같은 발견) |
| 새 도구/CLI 구축 | ✅ | 비슷한 도구의 설계 엿보기 |
| 도메인 특화 작업 (n8n, DB, SNS) | ❌ | SkillNet에 없음, 내부 지식이 우선 |
| 노트 정리/문서화 | ❌ | 우리 스킬이 이미 더 깊음 |
| 단순 작업/수정 | ❌ | 검색 비용 대비 이득 없음 |

### 프롬프트 템플릿

오케스트레이터가 스킬/도구 설계 작업을 위임할 때만 아래를 프롬프트에 추가:

```
시작 전 `skillnet search "{관련 키워드}" --mode keyword --limit 3`으로 
비슷한 스킬이 있는지 확인해. 평가 Good 3개+ 스킬만 다운로드해서 SKILL.md를 읽고,
구조적 패턴만 참고해. 범용 베스트프랙티스는 무시하고 우리 환경에 맞게 변형.
내부 지식(workcase, RPG, orchestration.db)이 항상 SkillNet보다 우선.
```

### 주의사항

- SkillNet 품질 들쭉날쭉 — 대부분 "범용 체크리스트" 수준. 구조적 패턴만 가치 있음
- 검색+다운로드+읽기에 subagent 턴 2-3회 소모 — 단순 작업에는 낭비
- Windows cp949 인코딩 문제 → 섹션 6의 `PYTHONIOENCODING=utf-8` 래핑 필요
- agent 타입: `general-purpose` 또는 `Explore` (Bash+Read 필요)

## Observations

### [pattern] 모드 분기 + 외부 참조 = 스킬의 라우터화
SKILL.md를 "라우터"로 전환 — 모드 판단 + 공통 규칙만 담고, 상세 절차는 references/에 분리. Activation 시 SKILL.md만 로딩(가볍게) → 모드 판단 → 필요한 reference만 추가 Read. 컨텍스트 비용이 모드 수에 비례하지 않고 O(1)이 됨. #skill-system #context-engineering

### [fact] PyTorch pr-review이 이 패턴의 실증 사례
97K stars 프로젝트에서 실사용 중. SKILL.md 204줄 + review-checklist.md + bc-guidelines.md로 분리. 3가지 모드(PR번호/브랜치/상세)가 같은 SKILL.md에서 분기. #reference

### [fact] 우리 29개 스킬 중 적용 가치 높은 것은 rpg(442줄)와 figma-workflow(172줄)
나머지는 단일 플로우(weekly-report, flow-sync), 모드 차이가 작거나(reference, workcase), 이미 잘 분리됨(metabase-dashboard). #pragmatic

### [method] 외부 참조 파일 = 선택적 컨텍스트 로딩
references/ 디렉토리의 파일은 SKILL.md에서 "참조: references/xxx.md" 형태로 언급. Claude가 해당 모드 진입 시에만 Read tool로 로딩. 전체 스킬 컨텍스트가 SKILL.md 줄 수에 비례하던 것을 모드당 필요량으로 전환. #context-engineering
