# 컴파일러와 프로그래밍 언어 계보 이해

> Claude C Compiler 프로젝트를 계기로 컴파일러, 인터프리터, 프로그래밍 언어 계보를 정리한 학습 기록

## 1. 전체 흐름

```
CPU (하드웨어)
  ▲
기계어 (010101...)        ← 사람이 직접 작성 (초기)
  ▲
어셈블리어                 ← 기계어를 읽기 쉽게
  ▲
최초의 C 컴파일러          ← 어셈블리어로 만듦 (1972)
  ▲
지금의 C 컴파일러          ← C로 자기 자신을 만듦 (부트스트래핑)
  ▲
Python, Rust, Go, Java 등  ← C/C++로 만듦
```

## 2. 핵심 개념

### C는 프로그래밍 언어의 라틴어

현대 컴퓨팅의 기반. 직접 쓰는 사람은 줄었지만 모든 곳의 밑바닥에 깔려 있다.

- Python 인터프리터 (CPython) → C로 만듦
- JavaScript 엔진 (V8) → C++로 만듦
- Ruby 인터프리터 → C로 만듦
- Java JVM → C/C++로 만듦

### 새로운 언어 = C의 단점 보완

```
C의 단점                    → 보완한 언어
메모리 관리를 사람이 직접     → Java (자동 관리)
메모리 버그가 위험함         → Rust (컴파일 때 차단)
문법이 어렵고 개발이 느림    → Python (쉽고 빠른 개발)
동시 처리가 복잡함           → Go (간단한 동시 처리)
```

"다른 철학의 언어"로 보일 수 있지만, 뒤집으면 전부 C의 부족한 점에서 출발한 것이다.

### 컴파일러 vs 인터프리터

```
컴파일러 (C, Rust, Go):
  소스코드 → [컴파일러] → .exe → CPU 직접 실행
  "책을 미리 번역해둔다" → 빠름, 통역 불필요

인터프리터 (Python, JavaScript):
  소스코드 → [인터프리터가 한 줄씩 실행] → CPU 실행
  "동시통역사가 옆에서 읽어준다" → 느림, 통역사 항상 필요
```

### 부트스트래핑

자기 자신으로 자기를 만드는 것.

1. 어셈블리어로 간단한 C 컴파일러 만듦
2. 그 C 컴파일러로 더 좋은 C 컴파일러를 C로 작성
3. 반복 → 지금의 GCC

Rust도 동일: 처음엔 다른 언어로 만들었지만 지금은 Rust 컴파일러가 Rust로 작성됨.

## 3. Claude C Compiler에서 얻은 해석

### "어려운 프로젝트를 해냈다"가 아니라 "소프트웨어 전체 스택을 관통하는 지식의 증명"

Claude가 C 컴파일러를 만들었다는 것은 단순히 복잡한 코드를 생성한 게 아니다. 고급 언어부터 기계어 명령어 체계까지, 소프트웨어의 전체 계층을 이해하고 있다는 의미다.

```
Claude가 알고 있는 범위:

인간 언어 (한국어, 영어)
    ↓
프로그래밍 언어 (Python, C, Rust)
    ↓
어셈블리어
    ↓
기계어 명령어 체계 (x86, ARM, RISC-V)  ← 여기까지
    ↓
전기 신호 (0/1)                         ← 물리적 영역, AI 관여 못함
```

다만 "규칙을 안다" ≠ "CPU처럼 실행할 수 있다". 영한사전을 만들 수 있는 사람이 영어 원어민은 아닌 것처럼, Claude는 변환 규칙을 체계적으로 아는 것이지 기계어로 사고하는 것은 아니다.

## Observations

- [fact] 현대 프로그래밍 언어 대부분의 런타임/인터프리터는 C/C++로 구현되어 있다 #foundation
- [pattern] 새로운 프로그래밍 언어는 C의 특정 단점을 보완하는 방향으로 탄생한다 #genealogy
- [fact] 부트스트래핑: 컴파일러가 자기 자신의 언어로 작성될 수 있다 (GCC, rustc) #bootstrapping
- [pattern] AI가 컴파일러를 만들 수 있다 = 소프트웨어 전체 스택에 대한 지식 체계를 보유한다는 증명 #ai-capability
- [warning] "규칙을 안다" ≠ "실행할 수 있다" — Claude는 변환 규칙을 아는 것이지 기계어로 사고하는 것은 아님 #nuance
