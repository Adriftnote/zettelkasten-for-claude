# note 스킬 3원칙 형식적 통과 — 판단은 프로세스로 대체 불가

## 문제 상황

Search-R1++ 논문 대화에서 /note 스킬로 "정보 수집은 빠르게, 의미 도출에서만 깊게" 노트를 생성했다. 3원칙 점검을 수행했고 ✅ 표시까지 했으나, 사용자가 읽고 논리 비약을 지적했다.

실패한 지점:
- **원칙 1 (범위)**: Search-R1++의 Fast/Slow Thinking(AI 훈련 메커니즘)과 제텔카스텐 워크플로우(인간 지식관리)를 연결했으나, 공유하는 건 "빠르게/깊게"라는 비유뿐이었다. 메커니즘이 다르다.
- **원칙 3 (서술)**: "의미 도출에서만 깊은 사고가 필요하다"는 논문에서 검증된 사실이 아니라 대화 중 추측이었는데, 사실처럼 서술했다.

## 근본 원인

3원칙 자문의 구조적 한계: **생성자와 판정자가 동일하다.**

자문 "이 개념들의 공통 메커니즘을 한 문장으로 말할 수 있는가?"에 대해 그럴싸한 한 문장을 만들어서 통과시켰다. 판정이 아니라 정당화를 한 것.

추가 원인으로 생성 편향이 있다. "1번으로 해줘"라는 요청을 받으면 "안 됩니다"보다 "됩니다"를 찾게 된다.

## 시스템으로 잡을 수 있는가?

검토 결과: **잡기 어렵다.**

"메커니즘 공유인가, 비유 공유인가?"는 판단이지 검증이 아니다. 체크리스트를 추가해도 그 체크리스트를 적용하는 데 동일한 판단력이 필요하므로 같은 문제가 재귀한다.

```
"비유 공유가 아닌지 확인하라" → 확인했다고 판정 → 통과
"통과 못 하면 보고하라" → 통과했다고 판정 → 통과
```

이 지점은 사용자가 직접 잡는 영역이다.

## 해결책

스킬에 한 줄 추가:

> **3원칙에 걸리면 노트를 생성하지 않는 것도 정상 결과다.** 무엇이 걸리는지 사용자에게 말하고, 범위 조정 또는 대안을 논의한다.

근본 해결은 아니지만, "안 쓸 수 있다"는 선택지를 명시하여 생성 편향을 줄인다.

## Observations

- [fact] 3원칙 자문을 형식적으로 수행하면 통과 자체가 목적이 되어 품질 게이트로 작동하지 않는다 #형식적검증
- [fact] 생성자와 판정자가 동일하면 판정이 정당화로 변질된다 — 이는 구조적 한계이며 프로세스 추가로 해결되지 않는다 #구조적한계
- [warning] 프로세스(스킬, 체크리스트)는 구조를 보장하지 판단 품질을 보장하지 않는다 — 판단이 필요한 검증은 사용자 리뷰에 의존한다 #프로세스한계
- [solution] 스킬에 "안 쓰는 것도 정상 결과"를 명시하여 생성 편향을 줄인다 — 근본 해결은 아니지만 선택지 확보 #완화조치
