```

```

vs 전체 파일 뒤지기

**6. Dual-view:**
```
의미(semantic) + 구조(topology)
RPG의 핵심 강점!

```
코드 구조 분석 도구
의존성 추출에 사용

## 🔤 Q13. 왜 "위상"이라는 단어를 써? 수학용어잖아?

**A:** 번역의 문제예요!

**진실:**
```
영어: Topological Order
     Topology = 구조, 배치, 연결 관계

한국어: 위상 순서
        위상 = 수학 용어 차용

→ 영어로는 평범한 단어!
→ 한국어로는 어려워 보임! 😅
```

---

**영어권에서는:**
```
Network Topology → "네트워크 구조"
Graph Topology → "그래프 구조"
Room Topology → "방 배치"

→ 그냥 일상적 단어!
→ 수학 떠올리지 않음!
```

---

**왜 Topology를 쓸까?**
```
수학에서:
"형태 바뀌어도 관계는 유지"

컴퓨터에서:
"코드 바뀌어도 구조는 유지"

→ 개념 차용!
→ 하지만 일반화됨
```

---

**당신의 이해 (정답!):**
```
✅ Topology = 영어로 그냥 "구조"
✅ 한국어 "위상" = 번역 과정에서 수학 용어 차용
✅ 영어권은 평범한 단어
✅ 한국인만 어렵게 느낌

→ 번역의 문제!
```

---

**읽을 때 팁:**
```
"위상 순서" 보면:
→ "구조 순서" 또는 "의존 순서"로 생각

"topology" 보면:
→ "구조" 또는 "배치"로 생각

→ 훨씬 이해 쉬워짐!
```

## 🏗️ Q14. 레포에서 자주 사용되는 구조가 정해져 있어? 클린아키텍처야?

**A:** 여러 개 있어요! 클린 아키텍처는 그 중 하나!

**📐 자주 쓰이는 구조들:**

**1️⃣ 계층 구조 (Layered)** - 가장 기본
```
├─ presentation/ (UI)
├─ business/ (로직)
├─ data/ (DB)
└─ infrastructure/ (설정)

→ 위에서 아래로만 의존!
```

---

**2️⃣ MVC 패턴** - 웹에서 많이 씀
```
├─ models/ (데이터)
├─ views/ (화면)
└─ controllers/ (중재자)

비유: 레스토랑
View = 메뉴판
Controller = 웨이터
Model = 주방
```

---

**3️⃣ 클린 아키텍처** ⭐
```
동심원 구조:
[외부: 프레임워크, DB]
  [중간: 어댑터]
    [안쪽: 비즈니스 로직]
      [중심: 핵심 규칙]

규칙: 의존성이 안쪽으로만!

장점:
✅ 핵심 로직 독립적
✅ DB 바꿔도 핵심 안 바뀜
✅ 테스트 쉬움
```

---

**4️⃣ 마이크로서비스**
```
큰 서비스를 작은 서비스로 쪼갬:
- user-service/
- product-service/
- order-service/

→ 각각 독립된 레포!
```

---

**5️⃣ 헥사고날 (Ports & Adapters)**
```
[핵심 로직]
    |
  Port (인터페이스)
    |
  Adapter (구현)
    |
  [DB/웹]

→ 어댑터만 바꾸면 됨!
```

---

**📏 프로젝트 크기별 적합한 구조:**

**🟢 아주 작음 (100줄):** 구조 필요 없음
```python
# main.py에 다!
```

**🟡 작음 (1,000줄):** 간단한 폴더 분리
```
├─ main.py
├─ utils.py
└─ data/
```

**🟠 중간 (1만 줄):** MVC 정도
```
├─ models/
├─ views/
└─ controllers/
```

**🔴 대형 (10만 줄+):** 클린 아키텍처
```
├─ domain/
├─ use_cases/
├─ adapters/
└─ infrastructure/
```

---

**좋은 구조의 공통 원칙: SOLID**
```
S: 단일 책임 (한 파일은 한 일만)
O: 개방/폐쇄 (확장은 열고, 수정은 닫기)
L: 리스코프 치환 (부모↔자식 교체 가능)
I: 인터페이스 분리 (필요한 것만)
D: 의존성 역전 (추상에 의존)
```

---

**RPG의 역할:**
```
RPG = 구조를 보이게 만드는 도구!

✅ 의존성 방향 시각화
✅ 순환 의존 자동 감지
✅ 구조 위반 경고
✅ 좋은 구조인지 한눈에!
```

## 😲 Q15. 클린아키텍처가 그중 하나라는 게 의외다. 모든 구조를 그렇게 짜야 하는 줄 알았는데?

**A:** 완전 정상적인 오해예요! 은탄환은 없어요!

**진실:**
```
클린 아키텍처:
✅ 대규모 프로젝트에 좋음
❌ 작은 프로젝트에는 과함!

비유:
고층 빌딩 설계도 (클린 아키텍처)
- 100층 빌딩: 필수!
- 단독 주택: 과함!
```

---

**📊 상황별 적합한 구조:**

**혼자, 빠르게:**
```
구조: 최소한!
예시: 개인 프로젝트, 프로토타입
```

**3-5명, 3개월:**
```
구조: MVC
예시: 스타트업 MVP
```

**10명+, 1년+:**
```
구조: 계층 또는 클린
예시: 기업 서비스
```

**100명+, 10년:**
```
구조: 클린 + 마이크로서비스
예시: 네이버, 카카오급
```

---

**클린 아키텍처의 단점:**

**1. 초기 개발 느림:**
```
간단한 구조: 로그인 1시간
클린 아키텍처: 로그인 3시간
→ 파일 5개 만들어야 함!
```

**2. 복잡함:**
```
간단: login.py (100줄, 1개 파일)
클린: 6개 파일 (170줄)
```

**3. 과한 추상화:**
```
비유: 숟가락 사용하기
- 인터페이스 정의
- 어댑터 구현
- 팩토리 패턴
→ 밥 먹으려다 배고파 죽음 😅
```

---

**🔄 구조는 진화해요:**
```
1단계 (1-3개월): 간단히
├─ app.py
└─ utils.py

2단계 (3-6개월): 정리
├─ models/
├─ views/
└─ controllers/

3단계 (6개월-1년): 구조화
├─ domain/
├─ use_cases/
└─ infrastructure/

4단계 (1년+): 분리
user-service/
product-service/

→ 나무처럼 자라면서 변화!
```

---

**유명한 조언:**
```
"Premature optimization is the root of all evil"
→ 조기 최적화는 모든 악의 근원

"YAGNI" (You Aren't Gonna Need It)
→ 지금 필요 없는 건 만들지 마

"KISS" (Keep It Simple, Stupid)
→ 단순하게 유지해!
```

---

**실제 사례: 페이스북**
```
2004: 단순한 PHP 파일들
2008: MVC 구조로 정리
2012: 여러 서비스로 분리
현재: 수천 개 마이크로서비스

→ 작게 시작해서 커짐!
```
