# 라이브러리와 프레임워크의 차이는 호출 방향 하나다

Module → Package → Library는 크기만 커지는 양적 변화다. Library → Framework는 호출 방향이 뒤집히는 질적 변화다. 이 질적 변화의 정체가 IoC(제어의 역전)다.

## 도출 근거

다음 facts의 조합에서 도출됨:

1. **Module은 파일, Package는 폴더, Library는 패키지 모음이다** — Module(파일 하나) → Package(폴더로 묶기) → Library(재사용 가능한 모음). 이 과정에서 변하는 건 "크기"뿐이다. 호출 방향은 항상 같다: 내가 코드를 호출한다.
   - 출처: module, package, library

2. **Framework는 내 코드를 호출한다** — 라이브러리는 `response = requests.get(url)` 처럼 내가 호출한다. 프레임워크는 `def index(request): return HttpResponse("Hello")` 처럼 내가 함수를 정의만 하면 프레임워크가 HTTP 요청 시 자동으로 호출한다.
   - 출처: framework

3. **이 뒤집힘의 정체가 IoC(Inversion of Control)다** — 일반 코드에서는 "내가 코드를 호출"하지만, IoC에서는 "시스템이 내 코드를 호출"한다. 프레임워크, 이벤트 핸들러, 의존성 주입, 콜백 함수 모두 IoC의 구현이다.
   - 출처: Inversion of Control (제어의 역전)

→ 따라서: **Module→Package→Library→Framework 순서를 "추상화 수준이 올라가는 사다리"로 보면 오해한다. Module→Library는 양적 확대(크기만 커짐)이고, Library→Framework는 질적 전환(호출 방향이 뒤집힘)이다. 이 질적 전환의 정체가 IoC이며, 이것 하나가 라이브러리와 프레임워크를 가른다.**

## 양적 변화 vs 질적 변화

```
Module ──→ Package ──→ Library ──→ Framework
  파일       폴더       모음         구조

[────── 양적 변화 ──────]  [── 질적 변화 ──]
크기만 커진다                호출 방향이 뒤집힌다
내가 호출한다               프레임워크가 호출한다
```

| 단계 | 변화 | 호출 방향 | 예시 |
|------|------|----------|------|
| Module → Package | 파일 → 폴더 | 내가 호출 | `import math_utils` |
| Package → Library | 폴더 → 배포 가능한 모음 | 내가 호출 | `import requests` |
| Library → Framework | 코드 모음 → 구조+규칙 | **프레임워크가 호출** | `def index(request):` |

처음 세 단계에서는 "이 코드를 어떻게 쓸지" 내가 결정한다. 프레임워크에서는 "내 코드를 언제 쓸지" 프레임워크가 결정한다.

## IoC의 실전 결과

호출 방향이 뒤집히면 나타나는 구체적인 차이:

| | 라이브러리 (내가 호출) | 프레임워크 (IoC) |
|---|---|---|
| **구조** | 자유롭게 설계 | 정해진 틀 따라야 함 |
| **규칙** | 권장사항 (무시 가능) | 필수 (안 따르면 작동 안 함) |
| **진입점** | 내가 만듦 | 프레임워크가 제공 |
| **디버깅** | 호출 스택 명확 | 호출 스택 복잡 |

라이브러리의 규칙은 무시해도 코드가 돌아간다. 프레임워크의 규칙을 무시하면 코드가 돌아가지 않는다. 이 차이가 IoC에서 온다.

## Observations

- [fact] Module→Package→Library는 크기만 커지는 양적 변화, Library→Framework는 호출 방향이 뒤집히는 질적 변화다 #abstraction-levels
- [fact] 라이브러리와 프레임워크를 가르는 건 IoC(제어의 역전) 하나다 — 누가 누구를 호출하느냐 #core-difference
- [fact] 라이브러리의 규칙은 무시해도 작동하지만, 프레임워크의 규칙은 무시하면 작동하지 않는다 — IoC가 강제하는 결과 #practical-consequence
- [method] "이 도구가 라이브러리인가 프레임워크인가?"를 판별할 때 "내가 호출하나, 시스템이 내 코드를 호출하나?"를 물으면 된다 #heuristic
