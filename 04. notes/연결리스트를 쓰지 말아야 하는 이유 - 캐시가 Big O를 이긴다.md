# 연결리스트를 쓰지 말아야 하는 이유 — 캐시가 Big O를 이긴다

배열과 연결리스트로 같은 순회를 돌리면 배열이 20배 빠르다. CPU 캐시 때문이다.

## 도출 근거

다음 facts의 조합에서 도출됨:

1. **1000만 개 숫자 합산 — 배열 10ms, 연결리스트 200ms** — 알고리즘은 동일(순차 순회 O(n)), 기계어도 동일, 데이터 배치만 다른데 20배 차이.
   - 출처: [[cache-friendliness]]

2. **L1 캐시 ~1ns, RAM ~100ns — 100배 속도 격차** — CPU가 데이터를 요청할 때, 캐시에 있으면 1ns, 없으면 100ns. 이 100배 격차가 알고리즘 복잡도보다 큰 변수가 된다.
   - 출처: [[Cache]]

3. **배열은 연속 메모리, 연결리스트는 흩어진 메모리** — CPU는 캐시 라인(64바이트)을 한 번에 로드한다. 배열은 다음 원소가 이미 캐시에 있지만, 연결리스트는 매번 RAM까지 가야 한다.
   - 출처: [[array]], [[linked-list]]

4. **Rust는 표준 라이브러리에 LinkedList를 넣고도 "쓰지 마라"고 권고한다** — `Vec`이나 `VecDeque`가 "거의 모든 경우" 더 빠르다는 게 공식 입장. 이론적으로 연결리스트가 유리한 삽입/삭제조차 캐시 효과 때문에 배열이 이기는 경우가 많다.
   - 출처: [[linked-list]] (Rust 비권장 섹션)

→ 따라서: **Big O는 "모든 메모리 접근이 동일 비용"이라고 가정하는데, 현실에서는 100배 차이가 난다. 이 가정이 깨지면 O(1) 삽입의 연결리스트가 O(n) 삽입의 배열에 진다. 자료구조 선택의 실전 기준은 "복잡도"가 아니라 "메모리 배치"다.**

## 왜 Big O가 틀리는가

Big O의 숨은 가정:

```
모든 메모리 접근 비용 = 상수 c

배열 순회:    n × c
리스트 순회:  n × c
→ 둘 다 O(n), "같다"
```

현실:

```
배열 순회:    n × 1ns   (캐시 히트 99%, L1에서 읽음)
리스트 순회:  n × 100ns (캐시 미스, 매번 RAM까지)
→ 둘 다 O(n)이지만, 상수 c가 100배 다르다
```

Big O가 버리는 상수 `c`에 캐시 계층의 100배 격차가 숨어 있다.

## 실전에서 뒤집히는 사례

| 연산 | Big O 승자 | 실전 승자 | 이유 |
|------|-----------|----------|------|
| 순차 순회 | 동률 O(n) | **배열** (20배) | 캐시 라인 프리페치 |
| 중간 삽입 | 연결리스트 O(1) | **배열** (대부분) | 리스트의 "위치 찾기" O(n) + 캐시 미스 |
| 앞쪽 삽입 | 연결리스트 O(1) | **VecDeque** | 링 버퍼로 O(1) + 캐시 친화 |
| 대량 삭제 | 연결리스트 O(1) | **경우에 따라** | 삭제 위치를 이미 알 때만 리스트 유리 |

연결리스트가 실전에서 이기는 유일한 조건: **삭제할 노드의 포인터를 이미 갖고 있고**, 데이터 크기가 캐시 라인(64B)보다 훨씬 커서 배열 이동 비용이 캐시 이점을 초과할 때.

## Rust의 설계 결정이 증명하는 것

Rust 표준 라이브러리의 선택:

| 용도 | 교과서 답 | Rust 권장 | 근거 |
|------|----------|----------|------|
| 동적 배열 | Array | `Vec<T>` | 연속 메모리, 캐시 최적 |
| 스택 | LinkedList | `Vec<T>` (push/pop) | 끝에서만 조작 → 배열 최적 |
| 큐 | LinkedList | `VecDeque<T>` | 링 버퍼, 양쪽 O(1) + 캐시 |
| 리스트 | LinkedList | `Vec<T>` | "대부분 Vec이 더 빠름" |

Rust가 메모리 안전성에 집착하는 언어라는 점을 고려하면, **LinkedList를 stdlib에 포함시키고도 비권장하는 것**은 이론과 실전의 괴리를 가장 극적으로 보여주는 사례다.

## Observations

- [fact] 같은 O(n) 순회에서 배열이 연결리스트보다 20배 빠르다 — Big O의 상수 c에 캐시 계층 100배 격차가 숨어있기 때문 #benchmark
- [fact] CPU 캐시 라인은 64바이트를 한 번에 적재한다 — 배열의 다음 원소는 이미 캐시에 있지만, 리스트의 다음 노드는 RAM 어딘가에 흩어져 있다 #mechanism
- [fact] Rust는 LinkedList를 표준 라이브러리에 넣고도 공식적으로 비권장한다 — 이론적 우위가 실전에서 성립하지 않음을 언어 차원에서 인정 #evidence
- [method] 자료구조 선택 시 "Big O 비교" 다음에 "메모리 배치 확인"을 추가해야 한다 — 연속 메모리 여부가 상수 100배를 결정 #heuristic
- [question] 캐시 라인 64B보다 원소 크기가 클 때 정확한 손익분기점은? — 원소 크기 × 이동 비용 > 캐시 미스 비용이 되는 지점 #follow-up
