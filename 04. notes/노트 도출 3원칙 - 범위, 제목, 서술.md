# 노트 도출 3원칙 - 범위, 제목, 서술

도출 노트를 쓸 때 범위, 제목, 서술 세 가지를 지킨다. 실패한 노트 2개와 성공한 노트 1개에서 도출.

## 도출 근거

다음 실패/성공 사례에서 도출됨:

1. **"캐시는 어디에나 있다"는 실패했다** - CPU 캐시, KV 캐시, Progressive Disclosure, 시스템1을 "캐시"로 묶었지만, 공유하는 건 "가까이 둔다"는 비유뿐이었다. 메커니즘은 전부 달랐다. "캐시의 보편성은 캐시 실패의 보편성을 의미한다"는 의미 없는 문장이었다.

2. **"Big O는 메모리를 모른다"는 제목이 틀렸다** - 내용은 연결리스트가 배열에 지는 이유인데, 제목은 추상적 주장이었다. 제목만 보고 무슨 얘긴지 알 수 없었다.

3. **"교과서에서 이렇게 가르친다"는 거짓이었다** - 본인이 그렇게 배운 적 없는데 그럴싸하게 쓴 것. 사실이 아닌 서술은 노트의 신뢰를 깨뜨린다.

4. **"연결리스트를 쓰지 말아야 하는 이유"는 성공했다** - 배열, 연결리스트, 캐시 라인이 하나의 물리적 원인(캐시 라인 64B 프리페치)으로 연결된다. 제목이 곧 내용이고, 벤치마크 20배 + Rust 공식 비권장이라는 사실만 썼다.

→ 따라서: **노트 품질은 세 가지로 결정된다 — 범위를 좁게 잡았는가, 제목이 내용과 같은가, 사실만 썼는가.**

## 3원칙

### 1. 범위 — 메커니즘이 같은 것끼리만

- 연결하려는 개념들이 하나의 원인을 공유해야 한다
- "이 둘이 같은 이유"를 한 문장으로 못 하면 억지 연결
- 표면적 비유 공유 ≠ 메커니즘 공유

### 2. 제목 — 읽으면 내용이 보여야 함

- 제목만 보고 "무슨 얘기인지" 바로 알 수 있어야 한다
- 추상적이거나 멋있어 보이는 제목은 내용과 어긋난다

### 3. 서술 — 사실만, 포장 없이

- 본인이 모르거나 경험하지 않은 것을 아는 척 쓰지 않는다
- 그럴싸하게 들리지만 의미 없는 문장은 쓰지 않는다
- 사실 + 원인, 끝

## Observations

- [method] 노트 작성 전 "이 개념들이 같은 이유를 한 문장으로 말할 수 있는가?" 자문한다 #범위검증
- [method] 제목을 쓴 뒤 "이 제목만 보고 내용을 알 수 있는가?" 확인한다 #제목검증
- [method] 서술을 쓴 뒤 "이게 사실인가, 포장인가?" 구분한다 #서술검증
- [fact] 4개 도메인 강제 통합보다 2-3개 concept의 좁은 연결이 더 나은 노트를 만든다 #실패사례
- [fact] 있어 보이는 문장은 의미가 없을 확률이 높다 #실패사례
