# 추상화는 3단 구조로 반복된다

소프트웨어의 모든 설계는 "복잡한 것 → 중간 계층 → 단순한 인터페이스"라는 동일한 3단 구조의 반복이다.

## 도출 근거

다음 facts의 조합에서 도출됨:

1. **추상화 패턴은 "직접 → 중간계층 → 간접" 3단 구조로 반복된다** - ORM, HTTP, 파일시스템, 프로그래밍 언어가 모두 같은 구조의 변형
2. **ORM은 DDL과 DML 모두 처리한다** - 클래스 정의 → DDL, 객체 조작 → DML로 변환. SQL이라는 복잡한 언어를 프로그래밍 객체로 추상화
3. **CS의 기초는 추상화다** - 복잡한 것을 단순한 인터페이스로 감추는 것이 컴퓨터 과학의 근본 원리

→ 따라서: **새로운 기술을 만날 때 "이것이 무엇을 추상화하는가?"를 찾으면 본질을 빠르게 이해할 수 있다.**

## 3단 구조 적용 사례

| 복잡한 것 | 중간 계층 (추상화) | 단순한 인터페이스 |
|-----------|-------------------|------------------|
| SQL (DDL+DML) | ORM | Python 클래스/객체 |
| 네트워크 소켓 | HTTP 프로토콜 | 요청/응답 |
| 디스크 섹터 | 파일시스템 (OS) | 파일 읽기/쓰기 |
| 기계어 (0/1) | 프로그래밍 언어 | 로직 작성 |
| 하드웨어 | 폰노이만 구조 | 프로그램 = 데이터 |

## 실천 방법

새로운 기술을 학습할 때 3가지를 묻는다:

1. **아래**: 이것이 감추고 있는 복잡한 것은 무엇인가?
2. **가운데**: 어떤 방식으로 추상화하는가? (매핑, 프로토콜, 인터페이스)
3. **위**: 사용자에게 노출되는 단순한 인터페이스는 무엇인가?

## Observations

- [method] 새 기술 학습 시 "무엇을 추상화하는가?"를 먼저 파악하면 본질을 빠르게 이해 #abstraction #learning
- [pattern] 모든 추상화는 "복잡 → 중간계층 → 단순" 3단 구조로 동일하다 #architecture
- [example] ORM: SQL(복잡) → ORM엔진(중간) → 클래스/객체(단순) #orm
- [insight] 하위 계층이 바뀌어도 상위는 영향 없음 — 느슨한 결합의 핵심 원리 #architecture
