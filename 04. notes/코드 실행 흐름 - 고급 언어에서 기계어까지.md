# 코드 실행 흐름: 고급 언어에서 기계어까지

코드 실행의 핵심: **모든 코드는 결국 0과 1이 되어 CPU에서 Fetch→Decode→Execute→Store 사이클을 탄다.**

## 도출 근거

다음 facts의 조합에서 도출됨:

1. **고급 언어는 추상화 수준이 높다** - 사람이 읽기 쉬움, 하드웨어 제어 간단
2. **기계어는 CPU가 직접 실행하는 0과 1이다** - 사람이 읽기 불가능
3. **컴파일러/인터프리터가 이 간극을 메운다** - 변환 방식의 차이
4. **CPU는 Fetch→Decode→Execute→Store를 초당 수십억 번 반복한다** - 3GHz = 30억 회/초

→ 따라서: **"추상화는 편의성과 속도의 트레이드오프"**
- 컴파일 언어 (C): 빠름, 플랫폼별 컴파일 필요
- 인터프리터 (Python): 느림 (25배), 즉시 실행
- 하이브리드 (Java): 중간 (1.8배), JIT로 점점 빨라짐

## Observations

- [fact] 모든 고급 언어는 결국 기계어(0과 1)로 변환된다 #execution
- [fact] CPU는 Fetch→Decode→Execute→Store 사이클을 반복한다 #cpu
- [fact] 3GHz CPU는 초당 30억 번 이 사이클을 수행한다 #performance
- [fact] 같은 코드도 CPU 아키텍처(x86, ARM)에 따라 다른 기계어가 된다 #platform
- [fact] Python은 C보다 약 25배 느리다 (인터프리터 vs 컴파일) #benchmark
- [fact] Java JIT는 자주 쓰는 코드를 기계어로 컴파일해서 재사용한다 #jit
- [method] 고급언어 → (컴파일러) → 어셈블리 → (어셈블러) → 기계어 → CPU #flow
- [method] 컴파일러 최적화: 반복문을 상수로 치환 (5×100 → 500) #optimization
- [example] C: main.c → gcc → main.exe → CPU 직접 실행 #compile
- [example] Python: script.py → 인터프리터 → 한 줄씩 번역하며 실행 #interpret
- [example] Java: Main.java → javac → Main.class(바이트코드) → JVM → JIT → CPU #hybrid
- [decision] -O0(디버깅), -O2(배포용), -O3(성능중요) 최적화 수준 선택 #optimization

## 전체 과정 개요

```
┌──────────────────────────────────────────────────────────┐
│                     개발자 작성                           │
│                  (사람이 읽을 수 있음)                     │
└──────────────────────────────────────────────────────────┘
                          ↓
┌──────────────────────────────────────────────────────────┐
│              고급 언어 (High-Level Language)              │
│                                                           │
│  Python:  print("Hello, World!")                         │
│  Java:    System.out.println("Hello, World!");           │
│  C:       printf("Hello, World!");                       │
└──────────────────────────────────────────────────────────┘
                          ↓
                  [컴파일러/인터프리터]
                          ↓
┌──────────────────────────────────────────────────────────┐
│           어셈블리어 (Assembly Language)                  │
│            (CPU 명령어의 영어식 표현)                      │
│                                                           │
│  MOV  AX, 5        ; AX 레지스터에 5를 넣어라             │
│  ADD  AX, 3        ; AX에 3을 더해라                      │
│  PUSH AX           ; 결과를 스택에 저장해라               │
└──────────────────────────────────────────────────────────┘
                          ↓
                     [어셈블러]
                          ↓
┌──────────────────────────────────────────────────────────┐
│                 기계어 (Machine Code)                     │
│                (CPU가 직접 실행하는 0과 1)                │
│                                                           │
│  10110000 00000101  (MOV AX, 5)                          │
│  00000100 00000011  (ADD AX, 3)                          │
│  01010000           (PUSH AX)                            │
└──────────────────────────────────────────────────────────┘
                          ↓
┌──────────────────────────────────────────────────────────┐
│                    CPU 실행                               │
│              (전기 신호로 변환되어 처리)                   │
└──────────────────────────────────────────────────────────┘
```

## 5.2 단계별 상세 설명

### 단계 1: 고급 언어 작성

**특징**:
- 사람이 이해하기 쉬운 영어 기반 문법
- 플랫폼 독립적 (어느 컴퓨터에서나 같은 코드)
- 추상화 수준이 높음 (복잡한 하드웨어 제어를 간단한 명령어로)

**예시 비교**:
```python
# Python (매우 높은 수준)
numbers = [1, 2, 3, 4, 5]
total = sum(numbers)
```

```c
// C (중간 수준)
int numbers[] = {1, 2, 3, 4, 5};
int total = 0;
for(int i = 0; i < 5; i++) {
    total += numbers[i];
}
```

**변환 방식**:

| 언어 유형 | 변환 방법 | 예시 |
|----------|----------|------|
| 컴파일 언어 | 한 번에 전체 변환 → 실행파일 생성 | C, C++, Go |
| 인터프리터 언어 | 실행 시 한 줄씩 변환 | Python, JavaScript |
| 하이브리드 | 중간 코드로 변환 후 실행 시 최종 변환 | Java, C# |

---

### 단계 2: 중간 단계 (선택적)

#### 2-1. 중간 표현 (Intermediate Representation)

일부 컴파일러는 중간 단계를 거칩니다.

```
고급 언어
    ↓
추상 구문 트리 (AST)
    ↓
중간 코드 (IR)
    ↓
최적화
    ↓
어셈블리어
```

**예시 (LLVM IR)**:
```llvm
; Python의 a = b + c를 LLVM IR로 표현
%1 = load i32, i32* %b
%2 = load i32, i32* %c
%3 = add i32 %1, %2
store i32 %3, i32* %a
```

#### 2-2. 바이트코드 (Bytecode)

Java, Python 같은 언어가 사용하는 중간 형태

```
Java 소스코드 (.java)
        ↓
    [javac 컴파일러]
        ↓
바이트코드 (.class)  ← 플랫폼 독립적
        ↓
      [JVM]
        ↓
    기계어 실행
```

**바이트코드 예시**:
```java
// Java 코드
int a = 5;
int b = 10;
int c = a + b;
```

```
// 바이트코드
bipush 5      // 5를 스택에 푸시
istore_1      // 변수 a에 저장
bipush 10     // 10을 스택에 푸시
istore_2      // 변수 b에 저장
iload_1       // a 로드
iload_2       // b 로드
iadd          // 더하기
istore_3      // c에 저장
```

### 단계 3: 어셈블리어 변환

**어셈블리어의 특징**:
- CPU 명령어를 사람이 읽을 수 있는 니모닉(mnemonic)으로 표현
- CPU 아키텍처별로 다름 (Intel x86, ARM, RISC-V 등)
- 레지스터(CPU 내부 저장소)를 직접 제어

**어셈블리어 명령어 예시**:

| 명령어 | 의미 | 예시 |
|--------|------|------|
| MOV | 데이터 이동 | `MOV AX, 5` (AX에 5 저장) |
| ADD | 덧셈 | `ADD AX, 3` (AX에 3 더하기) |
| SUB | 뺄셈 | `SUB BX, 2` |
| JMP | 점프 (이동) | `JMP LABEL` |
| CMP | 비교 | `CMP AX, 0` |
| CALL | 함수 호출 | `CALL PRINT` |

**실제 예시**:
```c
// C 코드
int add(int a, int b) {
    return a + b;
}
```

```asm
; x86-64 어셈블리어
add:
    push   rbp              ; 스택 프레임 설정
    mov    rbp, rsp
    mov    DWORD PTR [rbp-4], edi   ; a를 스택에 저장
    mov    DWORD PTR [rbp-8], esi   ; b를 스택에 저장
    mov    eax, DWORD PTR [rbp-4]   ; a를 eax에 로드
    add    eax, DWORD PTR [rbp-8]   ; b를 더함
    pop    rbp              ; 스택 프레임 정리
    ret                     ; 결과 반환
```

---

### 단계 4: 기계어 변환

**기계어의 특징**:
- 순수하게 0과 1로만 구성
- CPU가 직접 이해하고 실행
- 사람이 직접 읽기/쓰기 거의 불가능

**변환 과정**:
```
어셈블리어:  MOV AX, 5
                ↓
16진수:      B8 05 00
                ↓
2진수:       10111000 00000101 00000000
```

**기계어 구조**:
```
┌──────────┬──────────┬──────────┬──────────┐
│ 명령 코드 │  대상    │  소스    │  추가    │
│ (OpCode) │ 레지스터 │  값      │  정보    │
└──────────┴──────────┴──────────┴──────────┘
   10111000   00000101   00000000    ...
     ↑           ↑           ↑
  MOV 명령     AX 레지스터   값 5
```

**실제 기계어 예시**:
```
C 코드:         int x = 42;
어셈블리:       mov DWORD PTR [rbp-4], 42
16진수 기계어:  C7 45 FC 2A 00 00 00
2진수:          11000111 01000101 11111100 00101010 00000000 00000000 00000000
```

### 단계 5: CPU 실행

**CPU 내부 처리**:

```
1. Fetch (인출)
   ┌─────────────┐
   │   메모리    │
   │ 10110000... │ ← 명령어 가져오기
   └─────────────┘
          ↓
2. Decode (해석)
   ┌─────────────┐
   │  제어 장치  │
   │ "MOV AX, 5" │ ← 무엇을 할지 해석
   └─────────────┘
          ↓
3. Execute (실행)
   ┌─────────────┐
   │  ALU (연산) │
   │  AX ← 5     │ ← 실제 작업 수행
   └─────────────┘
          ↓
4. Store (저장)
   ┌─────────────┐
   │ 레지스터/메모리 │
   │  AX = 5     │ ← 결과 저장
   └─────────────┘
```

**Clock Cycle**:
현대 CPU는 1초에 수십억 번 이 과정 반복 (예: 3GHz = 초당 30억 번)

## 5.3 언어별 실행 흐름 비교

### 컴파일 언어 (C/C++)

```
개발 시점:
  main.c ──→ [컴파일러] ──→ main.exe
  (소스)      (gcc/clang)    (실행파일)

실행 시점:
  main.exe ──→ [CPU] ──→ 결과 출력
  (바로 기계어)   (직접 실행)
```

**장점**: 실행 속도 매우 빠름
**단점**: 플랫폼별로 각각 컴파일 필요 (Windows용, Mac용, Linux용)

---

### 인터프리터 언어 (Python)

```
개발 시점:
  script.py (소스코드만 작성)

실행 시점:
  script.py ──→ [Python 인터프리터] ──→ 결과 출력
                    ↓
              한 줄씩 번역하며 실행
              (바이트코드로 중간 변환)
```

**장점**: 작성 즉시 실행, 플랫폼 독립적
**단점**: 실행 속도 상대적으로 느림

---

### 하이브리드 (Java)

```
개발 시점:
  Main.java ──→ [javac] ──→ Main.class
  (소스)        (컴파일러)   (바이트코드)

실행 시점:
  Main.class ──→ [JVM] ──→ [JIT 컴파일] ──→ [CPU] ──→ 결과
  (바이트코드)   (가상머신)  (기계어 생성)   (실행)
```

**JIT (Just-In-Time) 컴파일**:
- 처음엔 인터프리터처럼 실행
- 자주 쓰는 코드는 기계어로 컴파일해서 재사용
- 점점 빨라짐 (Warm-up)

**장점**: 플랫폼 독립적 + 준수한 속도
**단점**: JVM 필요

## 5.4 실행 속도 비교

**같은 작업을 다른 언어로 수행했을 때 속도 비교**:

| 언어                  | 실행 시간 | 상대 속도   |
| ------------------- | ----- | ------- |
| C (컴파일)             | 1.0초  | 기준 (1배) |
| Rust (컴파일)          | 1.1초  | 1.1배    |
| Java (JIT)          | 1.8초  | 1.8배    |
| Go (컴파일)            | 2.0초  | 2배      |
| JavaScript (V8 JIT) | 3.5초  | 3.5배    |
| Python (인터프리터)      | 25초   | 25배     |

**참고**: 실제 속도는 작업 종류에 따라 크게 달라짐

## 5.5 최적화 (Optimization)

컴파일러는 코드를 기계어로 변환할 때 더 빠르게 실행되도록 최적화합니다.

**최적화 예시**:

```c
// 원본 코드
int result = 0;
for(int i = 0; i < 100; i++) {
    result += 5;
}
```

**최적화 후** (컴파일러가 자동으로):
```c
// 컴파일러가 알아서 간단하게 변환
int result = 500;  // 5 × 100 = 500
```

**최적화 수준**:

| 플래그 | 설명 | 속도 | 용도 |
|--------|------|------|------|
| -O0 | 최적화 안 함 | 느림 | 디버깅 |
| -O1 | 기본 최적화 | 보통 | 일반 개발 |
| -O2 | 적극 최적화 | 빠름 | 배포용 |
| -O3 | 최대 최적화 | 매우 빠름 | 성능 중요 |

## 5.6 플랫폼별 차이

**같은 코드, 다른 기계어**:

```c
// 같은 C 코드
int x = 10;
```

**Intel x86-64 기계어**:
```asm
mov DWORD PTR [rbp-4], 10
→ C7 45 FC 0A 00 00 00
```

**ARM 기계어**:
```asm
mov r0, #10
→ E3 A0 00 0A
```

**이유**: CPU 아키텍처가 다르면 명령어 세트도 다름

**크로스 컴파일**:
Windows에서 Linux용 실행 파일을 만들 수 있음 (교차 컴파일)
