# Task 분해 방법론 리서치 결과

**Task ID**: task-20260130-011  
**작성일**: 2026-01-30  
**목적**: Subagent에게 작업을 효과적으로 나누는 방법 조사

---

## 요약

Claude Code에서 Task를 효과적으로 분해하는 방법론을 Anthropic 공식 문서와 최신 연구를 기반으로 정리했습니다. 핵심은 **독립성, 병렬성, 측정 가능성**을 기준으로 작업을 나누고, **3가지 패턴(Pipeline, Parallel, Swarm)**을 상황에 맞게 적용하는 것입니다.

---

## 1. Anthropic 공식 정보

### 1.1 Built-in Subagent 종류

Claude Code는 3가지 내장 Subagent를 제공합니다:

| Subagent            | 모델         | 도구        | 용도                  |
| ------------------- | ---------- | --------- | ------------------- |
| **Explore**         | Haiku (빠름) | Read-only | 파일 검색, 코드 분석        |
| **Plan**            | 상속         | Read-only | Plan 모드에서 코드베이스 리서치 |
| **General-purpose** | 상속         | 모든 도구     | 복잡한 멀티스텝 작업         |

### 1.2 Subagent vs Task 선택 기준

**Main Conversation을 사용할 때:**
- 빈번한 질의응답이 필요한 경우
- 여러 단계가 맥락을 공유하는 경우
- 빠르고 간단한 변경 작업
- 지연시간이 중요한 경우

**Subagent를 사용할 때:**
- 대량의 출력이 발생하지만 요약만 필요한 경우
- 특정 도구 제한이나 권한 설정이 필요한 경우
- 자체 완결적이고 요약 결과만 반환하면 되는 경우

### 1.3 중요 제약 사항

**제약 1: Subagent는 다른 Subagent를 생성할 수 없음**
- 중첩된 계층 구조 불가
- 재귀적 작업 분해 불가
- Main Claude가 오케스트레이터 역할

**제약 2: 20k 토큰 오버헤드**
- Task와 Subagent 모두 실행 전 약 20,000 토큰의 컨텍스트 로딩 필요
- 병렬 처리는 최대 10개까지

**제약 3: Background Subagent 제한**
- 사전 승인된 권한만 사용 가능
- MCP 도구 사용 불가
- 질문 도구 호출 실패 시 계속 진행

## 2. Task 분해 3가지 패턴

### 2.1 Pipeline (순차적 의존성)

작업이 단계별로 진행되며 각 단계가 이전 단계에 의존하는 패턴입니다.

```
Research → Plan → Implement → Test
```

**적용 시점:**
- 멀티 단계 워크플로우
- 각 단계가 이전 결과를 입력으로 사용
- 예: 요구사항 분석 → 설계 → 구현 → 테스트

**TaskUpdate 예시:**
```
TaskCreate: task-1 (Research)
TaskCreate: task-2 (Plan)
TaskCreate: task-3 (Implement)

TaskUpdate:
  taskId: "2"
  addBlockedBy: ["1"]

TaskUpdate:
  taskId: "3"
  addBlockedBy: ["2"]
```

**장점:** 자동 의존성 관리, 명확한 단계별 진행
**단점:** 병렬 처리 불가, 전체 시간이 각 단계의 합

### 2.2 Parallel Specialists (병렬 전문가)

서로 독립적인 작업을 여러 전문 Agent가 동시에 수행하는 패턴입니다.

```
Security Review
Performance Analysis    (동시 실행)
Code Simplicity Check
```

**적용 시점:**
- 서로 독립적인 검토/분석 작업
- 다른 전문 영역
- 예: 보안, 성능, 코드 품질을 각각 다른 Agent가 리뷰

**장점:** 시간 절약, 전문성 활용
**단점:** 컨텍스트 소비, 10개 병렬 제한

### 2.3 Self-Organizing Swarms (자율 조직 스웜)

독립적인 작업 아이템을 생성하고 Worker들이 자율적으로 선택해서 수행하는 패턴입니다.

**적용 시점:**
- 동일한 유형의 작업이 많을 때
- 작업 간 의존성이 없을 때
- 예: 100개 파일 리뷰, 여러 API 엔드포인트 테스트

**장점:** 최대 병렬화, 유연한 작업 분배
**단점:** 조율 오버헤드, 작업 순서 보장 불가

## 3. Task 분해 5가지 기준

### 3.1 독립성 (Independence)
- 다른 작업과 최소한의 인터페이스만 필요한가?
- 다른 진행 중인 작업에 의존하지 않는가?
- 하나의 Control Account에 할당 가능한가?

### 3.2 병렬 가능성 (Parallelizability)
- 선행 작업의 결과 데이터 없이 시작 가능한가?
- 다른 작업과 동시에 실행 가능한가?

분해 전략:
- Coarse-grained: 큰 작업 몇 개 (오버헤드 최소화)
- Fine-grained: 작은 작업 많이 (병렬성 극대화)

### 3.3 측정 가능성 (Measurability)
- 시작/완료 날짜가 명확한가?
- 중간 마일스톤이 측정 가능한가?
- 완료 기준이 검증 가능한가?

### 3.4 작업 크기 (Task Size)
- 권장: 8~80시간
- Agent 관점: 20k 토큰 오버헤드 고려 시 의미 있는 작업 단위

### 3.5 전문성 매칭 (Specialization Match)

| 작업 유형 | 추천 Subagent | 이유 |
|----------|---------------|------|
| 코드베이스 탐색 | Explore (Haiku) | 빠르고 read-only |
| 코드 리뷰 | 커스텀 (read-only) | 수정 방지 |
| 복잡한 구현 | General-purpose | 모든 도구 필요 |
| DB 쿼리 검증 | 커스텀 (Bash + hook) | 읽기 전용 강제 |

## 5. 실전 적용 가이드

### 5.1 Task 분해 5단계

**Step 1: 작업 분석**
- 영역 파악
- 독립성 확인
- 병렬성 판단
- 패턴 선택

**Step 2: Task 생성**
```
TaskCreate:
  subject: "[task-001-a] 세부작업명"
  description: |
    ## 목표
    - 구체적 목표
    
    ## 담당
    worker-xxx
    
    ## 완료 조건
    - 측정 가능한 기준
```

**Step 3: 의존성 설정**
```
TaskUpdate:
  taskId: "task-001-c"
  addBlockedBy: ["task-001-a", "task-001-b"]
```

**Step 4: Subagent 할당**
```
Task:
  subagent_type: "worker-xxx"
  prompt: "작업 상세..."
```

**Step 5: 모니터링**
```
TaskList
TaskGet: "task-001-a"
```

### 5.2 피해야 할 안티패턴

**1. 과도한 분해**
❌ TaskCreate: "변수명 변경", "주석 추가", "포맷팅"
✅ TaskCreate: "코드 품질 개선 (변수명, 주석, 포맷팅)"

**2. 순환 의존성**
❌ Task A blockedBy [B], Task B blockedBy [A]
✅ 의존성 방향 명확히 정의

**3. 불명확한 완료 조건**
❌ description: "성능 개선"
✅ description: "Lighthouse 점수 90 이상, FCP < 1.5초"

## 7. 핵심 Takeaway

### Task 분해 3원칙
1. **독립성 우선** - 의존성 최소화, 병렬 처리 가능하게
2. **측정 가능** - 명확한 완료 조건, 검증 가능한 산출물
3. **적정 크기** - 너무 작지도 크지도 않게 (8-80시간)

### 패턴 선택

| 상황 | 패턴 | 예시 |
|------|------|------|
| 순차적 단계 | Pipeline | Research → Plan → Implement |
| 독립적 영역 | Parallel | 보안/성능/품질 리뷰 |
| 반복 작업 | Swarms | 100개 파일 분석 |

### 제약 사항
- Subagent는 Subagent 생성 불가 (1 레벨만)
- 20k 토큰 오버헤드 고려
- 병렬 처리 최대 10개
- Background Subagent는 MCP 도구 사용 불가
