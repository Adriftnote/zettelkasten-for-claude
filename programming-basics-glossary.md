# 프로그래밍 기초 용어집

> 비개발자를 위한 쉬운 프로그래밍 용어 해설서

---

## 1. 코드의 종류

### 1.1 소스코드 (Source Code)

**정의**: 사람이 읽고 쓸 수 있는 형태로 작성된 프로그램의 원본 텍스트

**비유**: 요리사가 작성한 레시피와 같습니다. 사람이 읽고 이해할 수 있는 언어로 작성되어 있습니다.

**예시**:
```python
# Python 소스코드 예시
name = "홍길동"
print(f"안녕하세요, {name}님!")
```

**특징**:
- `.py`, `.js`, `.java` 등의 확장자를 가짐
- 텍스트 에디터로 열어볼 수 있음
- 사람이 직접 수정 가능

---

### 1.2 스크립트 (Script)

**정의**: 별도의 컴파일 과정 없이 바로 실행되는 소스코드

**비유**: 배우의 대본과 같습니다. 대본을 보면서 바로 연기할 수 있듯이, 컴퓨터가 스크립트를 읽으면서 바로 실행합니다.

**예시**:
- 웹페이지의 JavaScript
- 자동화 작업을 위한 Python 스크립트
- Excel 매크로 (VBA)

**특징**:
- 작성 후 즉시 실행 가능
- 수정이 쉽고 빠름
- 주로 자동화, 웹페이지 동작에 사용

---

### 1.3 프로그램 (Program)

**정의**: 컴퓨터가 수행할 작업들의 명령어 집합

**비유**: 완성된 제품입니다. 한글(HWP), Chrome 브라우저, 카카오톡 같은 실행 파일들이 모두 프로그램입니다.

**종류**:
| 유형 | 설명 | 예시 |
|------|------|------|
| 응용 프로그램 | 사용자가 직접 사용 | MS Word, Photoshop |
| 시스템 프로그램 | 컴퓨터 운영 관리 | Windows, macOS |
| 유틸리티 프로그램 | 보조 기능 제공 | 압축 프로그램, 백신 |

---

### 1.4 바이너리 (Binary)

**정의**: 0과 1로만 이루어진 기계가 직접 실행할 수 있는 파일

**비유**: 완제품 약처럼 사람이 직접 분석하기 어렵지만 컴퓨터는 바로 이해할 수 있는 형태입니다.

**예시**:
- `.exe` 파일 (Windows 실행 파일)
- `.apk` 파일 (Android 앱)
- `.app` 파일 (Mac 애플리케이션)

**특징**:
- 텍스트 에디터로 열면 알 수 없는 기호들이 나옴
- 실행 속도가 매우 빠름
- 사람이 직접 수정 불가능

---

### 1.5 기계어 (Machine Language)

**정의**: CPU가 직접 이해하고 실행할 수 있는 0과 1의 명령어

**비유**: 뇌세포가 이해하는 전기 신호와 같습니다. 가장 근본적인 수준의 언어입니다.

**예시**:
```
10110000 01100001  // 기계어 (실제로는 이렇게 보임)
```

**단계별 비교표**:

| 단계 | 형태 | 읽는 주체 | 예시 |
|------|------|-----------|------|
| 고급 언어 | `print("Hello")` | 사람 | Python, Java |
| 어셈블리어 | `MOV AX, 1` | 프로그래머 | 어셈블리 |
| 기계어 | `10110000 01100001` | CPU | 바이너리 |

---

## 2. 실행 도구

### 2.1 컴파일러 (Compiler)

**정의**: 소스코드를 한 번에 기계어로 번역하는 프로그램

**비유**: 책 전체를 다른 언어로 번역한 후 출판하는 것과 같습니다.

**작동 과정**:
```
소스코드 (hello.c)
      ↓
  [컴파일러]
      ↓
실행 파일 (hello.exe)
```

**장점**:
- 실행 속도가 매우 빠름
- 번역이 끝나면 원본 소스코드 없이도 실행 가능
- 오류를 미리 발견할 수 있음

**단점**:
- 코드를 수정할 때마다 다시 컴파일 필요
- 운영체제별로 각각 컴파일 필요

**대표 언어**: C, C++, Go, Rust

---

### 2.2 인터프리터 (Interpreter)

**정의**: 소스코드를 한 줄씩 읽으면서 즉시 실행하는 프로그램

**비유**: 동시통역사가 말하는 즉시 번역하는 것과 같습니다.

**작동 과정**:
```
소스코드 (script.py)
      ↓
  [인터프리터] ← 한 줄씩 읽고 바로 실행
      ↓
    결과 출력
```

**장점**:
- 코드를 작성하고 즉시 실행 가능
- 수정이 쉽고 빠름
- 플랫폼 독립적 (어디서나 실행 가능)

**단점**:
- 컴파일된 프로그램보다 느림
- 실행할 때마다 소스코드 필요

**대표 언어**: Python, JavaScript, Ruby, PHP

---

### 2.3 런타임 (Runtime)

**정의**: 프로그램이 실행되는 환경 또는 실행되는 시점

**두 가지 의미**:

#### 의미 1: 실행 환경
프로그램을 실행시켜주는 소프트웨어 환경

**예시**:
- Java Runtime Environment (JRE) - Java 프로그램 실행 환경
- Node.js Runtime - JavaScript 실행 환경
- .NET Runtime - C# 프로그램 실행 환경

**비유**: 영화를 보려면 영화관(하드웨어) 외에도 영사기(런타임)가 필요한 것과 같습니다.

#### 의미 2: 실행 시점
프로그램이 실제로 작동하는 시간

**비교표**:
| 시점 | 설명 | 예시 |
|------|------|------|
| 컴파일 타임 | 코드를 번역하는 시점 | 문법 오류 발견 |
| 런타임 | 프로그램이 실행되는 시점 | 0으로 나누기 오류 발생 |

---

### 2.4 어셈블러 (Assembler)

**정의**: 어셈블리어를 기계어로 번역하는 프로그램

**비유**: 약어와 기호로 된 메모를 정식 문서로 변환하는 것과 같습니다.

**변환 과정**:
```
어셈블리어:  MOV AX, 5
              ↓
기계어:      10110000 00000101
```

**사용 사례**:
- 운영체제 핵심 부분 개발
- 임베디드 시스템 (가전제품, 자동차)
- 게임 엔진의 성능 중요 부분

---

### 2.5 링커 (Linker)

**정의**: 여러 개의 컴파일된 파일들을 하나의 실행 파일로 합치는 프로그램

**비유**: 여러 장의 퍼즐 조각을 맞춰서 완성된 그림을 만드는 것과 같습니다.

**작동 과정**:
```
main.o (메인 코드)
utils.o (유틸리티 함수)
graphics.o (그래픽 처리)
      ↓
   [링커]
      ↓
  program.exe (완성된 실행 파일)
```

---

### 2.6 가상 머신 (Virtual Machine)

**정의**: 실제 컴퓨터처럼 작동하는 소프트웨어 환경

**비유**: 컴퓨터 안에 또 다른 가상의 컴퓨터를 만드는 것과 같습니다.

**종류**:
| 유형 | 설명 | 예시 |
|------|------|------|
| 시스템 VM | 전체 운영체제를 가상화 | VMware, VirtualBox |
| 프로세스 VM | 특정 프로그램 실행 환경 | JVM (Java), Python VM |

**장점**:
- 한 번 작성하면 어디서나 실행 (Write Once, Run Anywhere)
- 안전한 샌드박스 환경 제공

## 3. 코드 묶음과 구조

### 3.1 모듈 (Module)

**정의**: 특정 기능을 수행하는 코드들을 하나의 파일로 묶은 것

**비유**: 레고 블록 하나와 같습니다. 작지만 특정 기능을 가진 독립적인 부품입니다.

**예시**:
```python
# math_utils.py (모듈)
def add(a, b):
    return a + b

def subtract(a, b):
    return a - b
```

```python
# main.py (모듈 사용)
import math_utils

result = math_utils.add(5, 3)  # 8
```

**특징**:
- 하나의 파일 = 하나의 모듈
- 재사용 가능
- 코드 정리 및 관리 용이

---

### 3.2 패키지 (Package)

**정의**: 여러 모듈을 디렉토리(폴더)로 구조화한 것

**비유**: 레고 세트처럼 관련된 블록들을 하나의 상자에 담은 것입니다.

**구조 예시**:
```
mypackage/
  ├── __init__.py
  ├── math/
  │   ├── __init__.py
  │   ├── basic.py      (덧셈, 뺄셈)
  │   └── advanced.py   (미적분, 통계)
  └── graphics/
      ├── __init__.py
      ├── 2d.py
      └── 3d.py
```

**사용법**:
```python
from mypackage.math import basic
from mypackage.graphics import 2d

basic.add(1, 2)
2d.draw_circle()
```

---

### 3.3 라이브러리 (Library)

**정의**: 특정 작업을 수행하기 위해 미리 만들어진 코드 모음

**비유**: 공구 세트와 같습니다. 필요할 때 꺼내 쓸 수 있는 도구 모음입니다.

**종류**:

| 유형 | 설명 | 예시 |
|------|------|------|
| 표준 라이브러리 | 언어에 기본 포함 | Python의 `os`, `sys` |
| 외부 라이브러리 | 별도 설치 필요 | `numpy`, `pandas` |
| 시스템 라이브러리 | 운영체제 제공 | Windows API, POSIX |

**대표 예시**:
- **NumPy**: 과학 계산 (행렬, 통계)
- **Requests**: HTTP 통신 (웹 데이터 가져오기)
- **Pillow**: 이미지 처리 (사진 편집)

**사용 예시**:
```python
import requests

# 웹페이지 내용 가져오기
response = requests.get("https://www.example.com")
print(response.text)
```

---

### 3.4 프레임워크 (Framework)

**정의**: 애플리케이션 개발을 위한 골격과 규칙을 제공하는 소프트웨어 구조

**비유**: 조립식 가구 세트와 같습니다. 틀과 규칙이 정해져 있어서 그 안에서 작업하면 됩니다.

**라이브러리 vs 프레임워크**:

| 구분 | 라이브러리 | 프레임워크 |
|------|-----------|-----------|
| 제어 | 개발자가 호출 | 프레임워크가 개발자 코드 호출 |
| 자유도 | 높음 | 규칙에 따라야 함 |
| 비유 | 공구 (내가 사용) | 공장 (정해진 절차 따름) |

**예시**:
```python
# Django 프레임워크 예시
# Django가 정한 규칙대로 코드 작성

# models.py (데이터 구조 정의)
class Article(models.Model):
    title = models.CharField(max_length=200)
    content = models.TextField()

# views.py (요청 처리)
def article_list(request):
    articles = Article.objects.all()
    return render(request, 'list.html', {'articles': articles})
```

**대표 프레임워크**:
- **웹**: Django (Python), Spring (Java), Express (JavaScript)
- **모바일**: Flutter, React Native
- **게임**: Unity, Unreal Engine

---

### 3.5 SDK (Software Development Kit)

**정의**: 특정 플랫폼을 위한 소프트웨어 개발 도구 모음

**비유**: 프라모델 제작 키트처럼 개발에 필요한 모든 것이 들어있는 세트입니다.

**포함 요소**:
- 컴파일러/인터프리터
- 라이브러리
- 문서 및 예제 코드
- 디버깅 도구
- 시뮬레이터/에뮬레이터

**예시**:
- **Android SDK**: 안드로이드 앱 개발
- **iOS SDK**: iPhone 앱 개발
- **AWS SDK**: Amazon Web Services 이용

---

### 3.6 API (Application Programming Interface)

**정의**: 소프트웨어끼리 소통할 수 있게 해주는 규칙과 도구

**비유**: 레스토랑의 메뉴판과 같습니다. 주문 방법과 받을 수 있는 음식이 정해져 있습니다.

**예시**:
```python
# 날씨 API 사용 예시
import requests

# API 호출 (주문)
response = requests.get(
    "https://api.weather.com/current",
    params={"city": "Seoul"}
)

# 응답 받기 (음식 받기)
weather_data = response.json()
print(weather_data['temperature'])  # 23도
```

**API 종류**:
| 유형 | 설명 | 예시 |
|------|------|------|
| 웹 API | 인터넷을 통한 통신 | 카카오 지도 API |
| 라이브러리 API | 코드 함수 호출 | Python `open()` 함수 |
| 운영체제 API | OS 기능 사용 | Windows API |

## 4. 컴퓨팅 역사의 선구자들

### 4.1 찰스 배비지 (Charles Babbage, 1791-1871)

**별명**: "컴퓨터의 아버지"

**주요 업적**:
- **차분 기관 (Difference Engine, 1822)**: 수학 계산표를 자동으로 만드는 기계
- **해석 기관 (Analytical Engine, 1837)**: 프로그래밍 가능한 최초의 범용 컴퓨터 설계

**혁신적인 아이디어**:
```
해석 기관의 구조 (현대 컴퓨터와 동일한 개념!)
┌──────────────────────────────────┐
│ Store (저장소)     → 메모리       │
│ Mill (공장)        → CPU          │
│ Input (펀치카드)   → 입력장치     │
│ Output (프린터)    → 출력장치     │
└──────────────────────────────────┘
```

**재미있는 사실**:
- 증기 기관으로 작동하는 컴퓨터를 구상
- 당시 기술로는 완성하지 못했지만, 설계는 완벽했음
- 2002년 런던 과학박물관이 설계도대로 제작했더니 정상 작동!

---

### 4.2 에이다 러브레이스 (Ada Lovelace, 1815-1852)

**별명**: "최초의 프로그래머"

**주요 업적**:
- 배비지의 해석 기관을 위한 **최초의 알고리즘** 작성
- 컴퓨터가 단순 계산을 넘어 음악, 그림 등을 만들 수 있을 것이라 예측

**작성한 프로그램**:
베르누이 수를 계산하는 알고리즘 (1843년)
```
Step 1: 변수 초기화
Step 2: 반복문 시작
Step 3: 계산 수행
Step 4: 결과 저장
Step 5: 조건 확인 후 Step 2로 또는 종료
```

**선견지명**:
> "해석 기관은 숫자뿐만 아니라 소리나 그림 같은 어떤 것도 처리할 수 있다"
> - 1843년, 범용 컴퓨팅 개념 제시

**영향**:
- 프로그래밍 언어 **Ada**는 그녀의 이름에서 유래
- 매년 10월 둘째 주 화요일은 "Ada Lovelace Day" (여성 과학기술인의 날)

---

### 4.3 앨런 튜링 (Alan Turing, 1912-1954)

**별명**: "인공지능의 아버지", "암호 해독의 영웅"

**주요 업적**:

#### 1. 튜링 기계 (Turing Machine, 1936)
모든 계산 가능한 문제를 풀 수 있는 이론적 컴퓨터 모델

**구조**:
```
[무한 테이프] ← → [읽기/쓰기 헤드] ← → [상태 기계]
   0 1 0 1 1 0 ...         ↑              규칙 테이블
```

**의미**: 현대 모든 컴퓨터의 이론적 기초

#### 2. 제2차 세계대전 암호 해독
- 독일군의 에니그마(Enigma) 암호 해독
- 전쟁을 2년 단축시켜 수백만 명의 생명을 구함
- 영화 <이미테이션 게임> (2014)의 주인공

#### 3. 튜링 테스트 (1950)

## 5. 코드 실행 흐름: 고급 언어에서 기계어까지

### 5.1 전체 과정 개요

```
┌──────────────────────────────────────────────────────────┐
│                     개발자 작성                           │
│                  (사람이 읽을 수 있음)                     │
└──────────────────────────────────────────────────────────┘
                          ↓
┌──────────────────────────────────────────────────────────┐
│              고급 언어 (High-Level Language)              │
│                                                           │
│  Python:  print("Hello, World!")                         │
│  Java:    System.out.println("Hello, World!");           │
│  C:       printf("Hello, World!");                       │
└──────────────────────────────────────────────────────────┘
                          ↓
                  [컴파일러/인터프리터]
                          ↓
┌──────────────────────────────────────────────────────────┐
│           어셈블리어 (Assembly Language)                  │
│            (CPU 명령어의 영어식 표현)                      │
│                                                           │
│  MOV  AX, 5        ; AX 레지스터에 5를 넣어라             │
│  ADD  AX, 3        ; AX에 3을 더해라                      │
│  PUSH AX           ; 결과를 스택에 저장해라               │
└──────────────────────────────────────────────────────────┘
                          ↓
                     [어셈블러]
                          ↓
┌──────────────────────────────────────────────────────────┐
│                 기계어 (Machine Code)                     │
│                (CPU가 직접 실행하는 0과 1)                │
│                                                           │
│  10110000 00000101  (MOV AX, 5)                          │
│  00000100 00000011  (ADD AX, 3)                          │
│  01010000           (PUSH AX)                            │
└──────────────────────────────────────────────────────────┘
                          ↓
┌──────────────────────────────────────────────────────────┐
│                    CPU 실행                               │
│              (전기 신호로 변환되어 처리)                   │
└──────────────────────────────────────────────────────────┘
```

---

### 5.2 단계별 상세 설명

#### 단계 1: 고급 언어 작성

**특징**:
- 사람이 이해하기 쉬운 영어 기반 문법
- 플랫폼 독립적 (어느 컴퓨터에서나 같은 코드)
- 추상화 수준이 높음 (복잡한 하드웨어 제어를 간단한 명령어로)

**예시 비교**:
```python
# Python (매우 높은 수준)
numbers = [1, 2, 3, 4, 5]
total = sum(numbers)
```

```c
// C (중간 수준)
int numbers[] = {1, 2, 3, 4, 5};
int total = 0;
for(int i = 0; i < 5; i++) {
    total += numbers[i];
}
```

**변환 방식**:
| 언어 유형 | 변환 방법 | 예시 |
|----------|----------|------|
| 컴파일 언어 | 한 번에 전체 변환 → 실행파일 생성 | C, C++, Go |
| 인터프리터 언어 | 실행 시 한 줄씩 변환 | Python, JavaScript |
| 하이브리드 | 중간 코드로 변환 후 실행 시 최종 변환 | Java, C# |

---

#### 단계 2: 중간 단계 (선택적)

**2-1. 중간 표현 (Intermediate Representation)**

일부 컴파일러는 중간 단계를 거칩니다.

```
고급 언어
    ↓
추상 구문 트리 (AST)
    ↓
중간 코드 (IR)
    ↓
최적화
    ↓
어셈블리어
```

**예시 (LLVM IR)**:
```llvm
; Python의 a = b + c를 LLVM IR로 표현
%1 = load i32, i32* %b
%2 = load i32, i32* %c
%3 = add i32 %1, %2
store i32 %3, i32* %a
```

**2-2. 바이트코드 (Bytecode)**

Java, Python 같은 언어가 사용하는 중간 형태

```
Java 소스코드 (.java)
        ↓
    [javac 컴파일러]
        ↓
바이트코드 (.class)  ← 플랫폼 독립적
        ↓
      [JVM]
        ↓
    기계어 실행
```

**바이트코드 예시**:
```java
// Java 코드
int a = 5;
int b = 10;
int c = a + b;
```

```
// 바이트코드
bipush 5      // 5를 스택에 푸시
istore_1      // 변수 a에 저장
bipush 10     // 10을 스택에 푸시
istore_2      // 변수 b에 저장
iload_1       // a 로드
iload_2       // b 로드
iadd          // 더하기
istore_3      // c에 저장
```

---

#### 단계 3: 어셈블리어 변환

**어셈블리어의 특징**:
- CPU 명령어를 사람이 읽을 수 있는 니모닉(mnemonic)으로 표현
- CPU 아키텍처별로 다름 (Intel x86, ARM, RISC-V 등)
- 레지스터(CPU 내부 저장소)를 직접 제어

**어셈블리어 명령어 예시**:

| 명령어 | 의미 | 예시 |
|--------|------|------|
| MOV | 데이터 이동 | `MOV AX, 5` (AX에 5 저장) |
| ADD | 덧셈 | `ADD AX, 3` (AX에 3 더하기) |
| SUB | 뺄셈 | `SUB BX, 2` |
| JMP | 점프 (이동) | `JMP LABEL` |
| CMP | 비교 | `CMP AX, 0` |
| CALL | 함수 호출 | `CALL PRINT` |

**실제 예시**:
```c
// C 코드
int add(int a, int b) {
    return a + b;
}
```

```asm
; x86-64 어셈블리어
add:
    push   rbp              ; 스택 프레임 설정
    mov    rbp, rsp
    mov    DWORD PTR [rbp-4], edi   ; a를 스택에 저장
    mov    DWORD PTR [rbp-8], esi   ; b를 스택에 저장
    mov    eax, DWORD PTR [rbp-4]   ; a를 eax에 로드
    add    eax, DWORD PTR [rbp-8]   ; b를 더함
    pop    rbp              ; 스택 프레임 정리
    ret                     ; 결과 반환
```

---

#### 단계 4: 기계어 변환

**기계어의 특징**:
- 순수하게 0과 1로만 구성
- CPU가 직접 이해하고 실행
- 사람이 직접 읽기/쓰기 거의 불가능

**변환 과정**:
```
어셈블리어:  MOV AX, 5
                ↓
16진수:      B8 05 00
                ↓
2진수:       10111000 00000101 00000000
```

**기계어 구조**:
```
┌──────────┬──────────┬──────────┬──────────┐
│ 명령 코드 │  대상    │  소스    │  추가    │
│ (OpCode) │ 레지스터 │  값      │  정보    │
└──────────┴──────────┴──────────┴──────────┘
   10111000   00000101   00000000    ...
     ↑           ↑           ↑
  MOV 명령     AX 레지스터   값 5
```

**실제 기계어 예시**:
```
C 코드:         int x = 42;
어셈블리:       mov DWORD PTR [rbp-4], 42
16진수 기계어:  C7 45 FC 2A 00 00 00
2진수:          11000111 01000101 11111100 00101010 00000000 00000000 00000000
```

---

#### 단계 5: CPU 실행

**CPU 내부 처리**:

```
1. Fetch (인출)
   ┌─────────────┐
   │   메모리    │
   │ 10110000... │ ← 명령어 가져오기
   └─────────────┘
          ↓
2. Decode (해석)
   ┌─────────────┐
   │  제어 장치  │
   │ "MOV AX, 5" │ ← 무엇을 할지 해석
   └─────────────┘
          ↓
3. Execute (실행)
   ┌─────────────┐
   │  ALU (연산) │
   │  AX ← 5     │ ← 실제 작업 수행
   └─────────────┘
          ↓
4. Store (저장)
   ┌─────────────┐
   │ 레지스터/메모리 │
   │  AX = 5     │ ← 결과 저장
   └─────────────┘
```

**Clock Cycle**:
현대 CPU는 1초에 수십억 번 이 과정 반복 (예: 3GHz = 초당 30억 번)

---

### 5.3 언어별 실행 흐름 비교

#### 컴파일 언어 (C/C++)

```
개발 시점:
  main.c ──→ [컴파일러] ──→ main.exe
  (소스)      (gcc/clang)    (실행파일)

실행 시점:
  main.exe ──→ [CPU] ──→ 결과 출력
  (바로 기계어)   (직접 실행)
```

**장점**: 실행 속도 매우 빠름
**단점**: 플랫폼별로 각각 컴파일 필요 (Windows용, Mac용, Linux용)

---

#### 인터프리터 언어 (Python)

```
개발 시점:
  script.py (소스코드만 작성)

실행 시점:
  script.py ──→ [Python 인터프리터] ──→ 결과 출력
                    ↓
              한 줄씩 번역하며 실행
              (바이트코드로 중간 변환)
```

**장점**: 작성 즉시 실행, 플랫폼 독립적
**단점**: 실행 속도 상대적으로 느림

---

#### 하이브리드 (Java)

```
개발 시점:
  Main.java ──→ [javac] ──→ Main.class
  (소스)        (컴파일러)   (바이트코드)

실행 시점:
  Main.class ──→ [JVM] ──→ [JIT 컴파일] ──→ [CPU] ──→ 결과
  (바이트코드)   (가상머신)  (기계어 생성)   (실행)
```

**JIT (Just-In-Time) 컴파일**:
- 처음엔 인터프리터처럼 실행
- 자주 쓰는 코드는 기계어로 컴파일해서 재사용
- 점점 빨라짐 (Warm-up)

**장점**: 플랫폼 독립적 + 준수한 속도
**단점**: JVM 필요

---

### 5.4 실행 속도 비교

**같은 작업을 다른 언어로 수행했을 때 속도 비교**:

| 언어 | 실행 시간 | 상대 속도 |
|------|----------|----------|
| C (컴파일) | 1.0초 | 기준 (1배) |
| Rust (컴파일) | 1.1초 | 1.1배 |
| Java (JIT) | 1.8초 | 1.8배 |
| Go (컴파일) | 2.0초 | 2배 |
| JavaScript (V8 JIT) | 3.5초 | 3.5배 |
| Python (인터프리터) | 25초 | 25배 |

**참고**: 실제 속도는 작업 종류에 따라 크게 달라짐

---

### 5.5 최적화 (Optimization)

컴파일러는 코드를 기계어로 변환할 때 더 빠르게 실행되도록 최적화합니다.

**최적화 예시**:

```c
// 원본 코드
int result = 0;
for(int i = 0; i < 100; i++) {
    result += 5;
}
```

**최적화 후** (컴파일러가 자동으로):
```c
// 컴파일러가 알아서 간단하게 변환
int result = 500;  // 5 × 100 = 500
```

**최적화 수준**:
| 플래그 | 설명 | 속도 | 용도 |
|--------|------|------|------|
| -O0 | 최적화 안 함 | 느림 | 디버깅 |
| -O1 | 기본 최적화 | 보통 | 일반 개발 |
| -O2 | 적극 최적화 | 빠름 | 배포용 |
| -O3 | 최대 최적화 | 매우 빠름 | 성능 중요 |

### 5.6 플랫폼별 차이

**같은 코드, 다른 기계어**:

```c
// 같은 C 코드
int x = 10;
```

**Intel x86-64 기계어**:
```asm
mov DWORD PTR [rbp-4], 10
→ C7 45 FC 0A 00 00 00
```

**ARM 기계어**:
```asm
mov r0, #10
→ E3 A0 00 0A
```

**이유**: CPU 아키텍처가 다르면 명령어 세트도 다름

**크로스 컴파일**:
Windows에서 Linux용 실행 파일을 만들 수 있음 (교차 컴파일)

## 8. 자주 묻는 질문 (FAQ)

**Q1: 프로그래밍 언어는 몇 개나 있나요?**
A: 약 700개 이상! 하지만 실제로 많이 쓰이는 건 20~30개 정도입니다.

**Q2: 어떤 언어를 먼저 배워야 하나요?**
A: 목적에 따라 다릅니다.
- 입문: Python (쉽고 다용도)
- 웹: JavaScript
- 앱: Swift (iOS), Kotlin (Android)
- 게임: C#, C++

**Q3: 컴파일 언어가 인터프리터 언어보다 항상 빠른가요?**
A: 대부분 그렇지만, JIT 컴파일러를 쓰는 언어(Java, JavaScript V8)는 거의 비슷한 속도를 냅니다.

**Q4: 기계어를 직접 작성할 수 있나요?**
A: 이론적으로는 가능하지만, 현실적으로는 너무 복잡합니다. 초기 컴퓨터 개발자들도 곧바로 어셈블리어로 넘어갔습니다.

**Q5: AI가 프로그래머를 대체할까요?**
A: AI는 도구일 뿐, 프로그래머의 역할은 더 중요해질 것입니다. AI에게 무엇을 만들지 지시하고, 결과를 검증하는 능력이 핵심이 됩니다.
