# 캐시 친화성 (Cache Friendliness)

## 핵심 정의

캐시 친화성은 데이터가 CPU 캐시를 얼마나 효율적으로 활용하는지를 나타냅니다.
같은 알고리즘, 같은 기계어라도 데이터 배치에 따라 성능이 수십 배 차이날 수 있습니다.

## 실험: 1000만 개 숫자 더하기

### 배열 (캐시 친화적)
```rust
let arr = vec![1, 2, 3, ...];  // 연속 배치
// → 10ms (캐시 히트율 99%)
```

### 연결 리스트 (캐시 비친화적)
```rust
let list = LinkedList::from([1, 2, 3, ...]);  // 흩어진 배치
// → 200ms (캐시 미스 많음)
```

→ **20배 차이!** (같은 알고리즘, 같은 기계어)

## 왜 배열이 빠른가?

CPU가 1000번지를 읽으면...

```
캐시에 자동으로 로드:
[1000][1001][1002]...[1063] (64바이트)

다음 접근 시 캐시에서 읽음 → 초고속!
```

이것이 바로 캐시 라인(Cache Line)의 개념입니다. CPU는 메모리를 한 바이트 단위가 아닌 캐시 라인 단위로 로드하기 때문에, 연속된 메모리에 접근할 때 캐시 히트율이 극도로 높아집니다.

## 캐시 히트 vs 캐시 미스

| 상황 | 시간 | 비유 |
|------|------|------|
| L1 캐시 히트 | ~1ns | 책상 위 |
| L2 캐시 히트 | ~4ns | 서랍 안 |
| L3 캐시 히트 | ~12ns | 방 안 캐비닛 |
| RAM 접근 | ~100ns | 창고 |

→ RAM은 L1보다 **100배 느림**!

이는 현대 CPU 설계의 가장 큰 성능 격차입니다. 따라서 캐시 효율성을 고려하지 않은 코드는 예상 밖의 성능 저하를 겪을 수 있습니다.

## 캐시 친화적 vs 비친화적

| 특성 | 캐시 친화적 | 캐시 비친화적 |
|------|------------|--------------|
| 메모리 배치 | 연속 | 흩어짐 |
| 접근 패턴 | 순차적 | 무작위 점프 |
| 예측 가능성 | 높음 (prefetch) | 낮음 |
| 대표 구조 | 배열, Vec | 연결 리스트, 트리 |

## 실전 최적화 팁

### 1. 배열 우선
가능하면 Vec 사용. 동적 배열은 연속 메모리 할당으로 캐시 효율성을 보장합니다.

### 2. 연속 접근
순차 반복이 무작위 접근보다 훨씬 빠릅니다.

### 3. 구조체 배열 vs 배열 구조체
- **AoS (Array of Structures)**: 각 구조체가 흩어져 있음
- **SoA (Structure of Arrays)**: 같은 필드끼리 모여있음 (캐시 효율적)

### 4. 핫 데이터 분리
자주 접근하는 필드를 같은 캐시 라인에 배치합니다.

## Rust에서의 적용

### 좋은 예시: Vec (연속 메모리)
```rust
let data: Vec<i32> = vec![1, 2, 3, 4, 5];
```

### 피해야 할 예시: LinkedList (흩어진 메모리)
```rust
let list: LinkedList<i32> = LinkedList::new();
```

### 좋은 예시: 순차 접근
```rust
for item in &data {
    // 처리
}
```

### 피해야 할 예시: 무작위 접근
```rust
for i in random_indices {
    data[i];  // 캐시 미스 많음
}
```
