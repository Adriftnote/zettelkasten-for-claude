# 풋프린트 (Footprint)

소프트웨어가 차지하는 자원(디스크, 메모리, CPU 등)의 크기

## 📖 개요

풋프린트는 프로그램이 시스템 자원을 얼마나 "밟고 서 있는가"를 나타내는 지표다. 발자국(footprint)이 작을수록 가볍고 효율적이라는 비유에서 온 용어이다.

주로 메모리 풋프린트(실행 시 메모리 점유량)와 디스크 풋프린트(설치 용량)를 가리키며, 임베디드/IoT/엣지 환경처럼 자원이 제한된 곳에서 특히 중요한 설계 기준이 된다.

## 🎭 비유

캠핑 장비와 같다. 배낭여행(임베디드)에는 초경량 텐트(작은 풋프린트)가 필요하고, 글램핑(서버)에는 대형 텐트(큰 풋프린트)도 괜찮다. 같은 "잠자리" 기능이지만, 상황에 따라 적정 크기가 다르다.

## ✨ 특징

- **메모리 풋프린트**: 프로그램 실행 시 RAM을 얼마나 차지하는가
- **디스크 풋프린트**: 설치/배포 시 저장 공간을 얼마나 차지하는가
- **트레이드오프 관계**: 풋프린트가 작으면 보통 기능이나 성능에서 타협이 생김
- **환경 의존적 중요도**: 서버에서는 덜 중요하지만, IoT/임베디드에서는 핵심 제약 조건

## 💡 예시

### JS 런타임별 풋프린트 비교

| 런타임 | 바이너리 크기 | 풋프린트 등급 |
|--------|-------------|-------------|
| Node.js | ~100MB+ | 대 |
| Deno | ~100MB+ | 대 |
| Bun | ~50MB+ | 중 |
| txiki.js | 수 MB | **극소** |

txiki.js가 "최소 풋프린트"를 강조하는 이유: V8(JIT 컴파일러 포함) 대신 QuickJS-ng(순수 인터프리터)를 택해서 바이너리 크기 자체를 수십 분의 1로 줄인 것.

### 풋프린트의 다양한 맥락

```
Docker 이미지 풋프린트:
  alpine:latest   →   5MB  (작은 풋프린트)
  ubuntu:latest   →  77MB  (보통)

앱 번들 풋프린트:
  React 기본 번들  →  ~40KB gzipped
  jQuery          →  ~30KB gzipped
```
