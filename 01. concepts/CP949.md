# CP949

## 정의

**CP949 (Code Page 949)**는 Microsoft가 만든 한글 [[Character Encoding]]으로, 한국어 Windows의 기본 인코딩입니다.

**별칭:** 
- Windows-949
- MS949
- UHC (Unified Hangul Code, 통합 완성형)

## 특징

```
1 byte: ASCII (0x00 ~ 0x7F)
2 bytes: 한글 및 한자 (0x80 ~)

총 문자 수: 약 17,000개
- 한글: 11,172자 (완성형)
- 한자: 약 4,800자
- 기타 기호
```

## 구조

| 범위 | 바이트 | 내용 |
|------|--------|------|
| 0x00~0x7F | 1 byte | [[ASCII]] (영어, 숫자, 기호) |
| 0x81~0xFE + 0x41~0xFE | 2 bytes | 한글 음절 |
| 0xA1~0xFE + 0xA1~0xFE | 2 bytes | 한자, 기호 |

## 예시

### 한글 '가'

```
Unicode: U+AC00
CP949:   0xB0 0xA1 (2 bytes)
```

### 한글 '한'

```
Unicode: U+D55C
CP949:   0xC7 0xD1 (2 bytes)
```

### 영어 'A'

```
ASCII/Unicode: U+0041 (65)
CP949:         0x41 (1 byte, ASCII와 동일)
```

## 역사

| 시기 | 내용 |
|------|------|
| **1987** | KS X 1001 (완성형 한글 2,350자) |
| **1990년대 초** | Microsoft가 CP949 개발 |
| **1995** | Windows 95 한글판에 탑재 |
| **2000년대** | [[UTF-8]] 대두로 쇠퇴 |
| **현재** | 레거시 시스템에만 사용 |

## 한계

### 1. 한국어 전용

```
✅ 가능: 한글, 한자, 영어
❌ 불가능: 일본어, 중국어, 이모지

다국어 문서 작성 불가능!
```

### 2. 모든 한글 조합 불가

```
가능: 11,172개 완성형 한글
불가능: "똠", "쀍" 같은 일부 조합

→ 이름에 쓸 수 없는 글자 존재
```

### 3. 정렬 문제

```
CP949 바이트 순서 ≠ 가나다 순서

예: "가각간"을 바이트로 정렬하면 순서가 틀림
→ 올바른 정렬은 별도 처리 필요
```

### 4. 플랫폼 종속

```
Windows: CP949
Linux/Mac: UTF-8

→ 파일 교환 시 깨짐 문제
```

## 인코딩 깨짐 예시

### UTF-8 → CP949 오독

```
"그룹" (UTF-8):
  EA B7 B8 EB A3 B9

CP949로 잘못 읽으면:
  洹몃９ (이상한 한자)
```

→ [[PowerShell 한글 인코딩 문제]]

### CP949 → UTF-8 오독

```
"가" (CP949):
  B0 A1

UTF-8로 잘못 읽으면:
  � � (물음표, 깨진 문자)
```

## Windows에서의 역할

### 시스템 코드페이지

```
제어판 → 지역 → 시스템 로캘
→ "한국어(대한민국)"
→ 시스템 코드페이지 = CP949
```

**영향:**
- 콘솔 출력 기본 인코딩
- 파일명 인코딩
- 레거시 프로그램 문자 처리

### PowerShell 5.x 문제

```
PowerShell 5.x 기본값:
- 파일 읽기: CP949 (BOM 없으면)
- 파일 쓰기: UTF-8 (BOM 있음)

→ 불일치로 깨짐 발생!
```

→ [[BOM (Byte Order Mark)]] 참조

## CP949 vs EUC-KR

| 항목 | EUC-KR | CP949 |
|------|--------|-------|
| **표준** | 한국 산업 표준 | Microsoft 확장 |
| **한글 수** | 2,350자 | 11,172자 |
| **호환성** | Unix/Linux | Windows |
| **별칭** | KS X 1001 | UHC, MS949 |

**관계:**
```
EUC-KR ⊂ CP949

CP949는 EUC-KR의 상위 집합
→ CP949가 더 많은 한글 포함
```

## UTF-8로 전환해야 하는 이유

| 문제 | CP949 | UTF-8 |
|------|-------|-------|
| **다국어** | ❌ 불가능 | ✅ 가능 |
| **플랫폼** | Windows만 | 모든 OS |
| **웹 표준** | ❌ 비표준 | ✅ 표준 (98%) |
| **이모지** | ❌ 불가능 | ✅ 가능 |
| **미래** | 레거시 | 현대 표준 |

## 실용 가이드

### 인코딩 감지 (Python)

```python
import chardet

with open('file.txt', 'rb') as f:
    raw = f.read()
    result = chardet.detect(raw)
    print(result)  # {'encoding': 'CP949', 'confidence': 0.99}
```

### CP949 → UTF-8 변환

```python
# 파일 읽기 (CP949)
with open('old.txt', encoding='cp949') as f:
    text = f.read()

# 파일 쓰기 (UTF-8)
with open('new.txt', 'w', encoding='utf-8') as f:
    f.write(text)
```

### Windows 콘솔 출력

```python
# Windows 콘솔은 CP949
print("한글")  # 잘 나옴

# UTF-8로 리디렉션 시
with open('out.txt', 'w', encoding='utf-8') as f:
    print("한글", file=f)  # 명시적 인코딩 필요
```

## 디버깅

### 바이트 확인

```python
text = "가"

# CP949 바이트
bytes_cp949 = text.encode('cp949')
print(bytes_cp949.hex())  # 'b0a1'

# UTF-8 바이트
bytes_utf8 = text.encode('utf-8')
print(bytes_utf8.hex())  # 'eab080'
```

### 깨진 문자 복구

```python
# 잘못된 디코딩으로 깨진 문자열
broken = "洹몃９"

# UTF-8 바이트를 CP949로 읽은 경우
# 1. 원래 바이트 복구
raw_bytes = broken.encode('cp949')

# 2. UTF-8로 다시 디코딩
fixed = raw_bytes.decode('utf-8')
print(fixed)  # "그룹"
```

## 주의사항

### 1. 인코딩 명시

```python
# ❌ 위험: 시스템 기본 인코딩 (Windows: CP949)
with open('file.txt') as f:
    text = f.read()

# ✅ 안전: 명시적 인코딩
with open('file.txt', encoding='utf-8') as f:
    text = f.read()
```

### 2. 웹에서 절대 사용 금지

```html
<!-- ❌ 잘못됨 -->
<meta charset="CP949">

<!-- ✅ 올바름 -->
<meta charset="UTF-8">
```

### 3. 새 프로젝트는 UTF-8

```
레거시 유지보수: CP949 유지 가능
새 프로젝트: 무조건 UTF-8

→ CP949는 기술 부채!
```

## 핵심 요약

CP949는 "한국 Windows의 레거시 인코딩"입니다. 1990년대에는 유용했지만, 다국어 지원 불가, 플랫폼 종속 등의 한계로 현재는 [[UTF-8]]로 대체되었습니다. **새 프로젝트는 무조건 UTF-8을 사용**하세요!
