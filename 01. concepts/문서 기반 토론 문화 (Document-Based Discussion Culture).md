# 문서 기반 토론 문화 (Document-Based Discussion Culture)

"말로 하지 말고, 써서 내고, 써서 토론하라" — 제안·토론·결정 전 과정을 문서로 수행하는 의사결정 방식.

## 📖 개요

문서 기반 토론 문화는 중요한 의사결정을 구두 회의가 아닌 문서를 중심으로 진행하는 방식이다. 세 가지 원칙으로 구성된다: (1) 제안을 문서로 쓴다, (2) 토론도 문서로 한다, (3) 결정 사유도 문서로 남긴다. 인터넷 표준화 기구(IETF)의 RFC(1969)에서 시작되어 오픈소스 커뮤니티를 거쳐 테크 업계 전반으로 확산된 관행이다.

## 🎭 비유

국회의 법안 발의 과정과 유사하다. 누군가 법안(제안서)을 문서로 작성하고, 위원회(커뮤니티)에서 공개 토론한 뒤, 찬반 사유를 기록과 함께 채택하거나 거부한다. 구두 합의만으로 법이 되지 않듯, 코드 변경도 문서 없이 진행하지 않는다.

## ✨ 특징

- **투명성**: 제안과 토론 과정이 기록으로 남아 누구나 추적 가능
- **비동기 참여**: 같은 시간·장소에 있지 않아도 토론에 참여 가능
- **맥락 보존**: "왜 이렇게 결정했는가"가 6개월, 6년 뒤에도 확인 가능
- **품질 필터**: 문서로 쓰는 행위 자체가 아이디어를 정제하는 효과
- **신규 참여자 온보딩**: 과거 결정의 맥락을 스스로 학습할 수 있음

## 💡 예시

업계에서 사용되는 주요 구현체:

| 이름 | 영역 | 시작 | 특징 |
|------|------|------|------|
| RFC | 인터넷 표준 (IETF) | 1969 | 완전 공개, 누구나 제안 |
| PEP | Python | 2000 | 공개 토론, 커뮤니티 합의 |
| Design Doc | Google 내부 | — | 사내 공개, 동료 리뷰 |
| 6-Pager / PR FAQ | Amazon | — | 회의 시작 시 전원이 현장에서 묵독 후 토론 |
| ADR | 소프트웨어 아키텍처 | 2011 | 아키텍처 결정 기록 |
| JEP | Java | 2011 | OpenJDK 변경 제안 |
| Rust RFC | Rust | 2014 | 공개 토론, 누구나 제안 |

적용 방식의 스펙트럼 — 같은 원칙이 동기(synchronous)와 비동기(asynchronous) 양쪽으로 구현된다:

```
생각을 말로 전달        글로 전달 + 말로 토론            전부 글로
───────────────────────────────────────────────────────────
PPT 회의               Amazon 6-Pager                  PEP/RFC
```

Amazon 6-Pager의 핵심 규칙: "미리 읽어오지 말고 그 자리에서 읽어라." 미리 읽어오라고 하면 현실적으로 안 읽고 오기 때문에, 회의 첫 20~30분을 전원 침묵 속 묵독(study hall)으로 강제한다. "다 읽었다"는 전제를 희망이 아니라 구조로 보장하는 방식.

공개 수준의 스펙트럼:

```
완전 공개                                     완전 폐쇄
←──────────────────────────────────────────────→
PEP/RFC       Google Design Doc     Apple      전통적 SI
(누구나 제안)    (사내 공개)          (극비 silo)  (top-down)
```
